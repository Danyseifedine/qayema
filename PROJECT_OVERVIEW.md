# Qayema — Project Overview

> **Qayema** is a Laravel 12 + Filament v4 + Livewire 3 multi-tenant SaaS for QR-code digital restaurant menus.
> Restaurant owners build multi-language menus in a custom dashboard; visitors scan a QR code to view a mobile-first public menu;
> admins manage everything from a Filament back office. Includes AI menu scanning (Google Gemini), visitor analytics, and a security/hardening layer.

This document is a comprehensive, code-derived map of the entire application. It was generated by reading the actual source of every
subsystem (models, migrations, controllers, services, Filament resources, middleware, views, config, and tests). The legacy `PROJECT_PLAN.md`
describes an older "MenuX" vision and is superseded by this overview where they differ — the code is the source of truth.

## 1. Overview & Architecture

### 1.1 What the product is

**qayema** is a multi-tenant SaaS that lets restaurants publish QR-code-driven digital menus. A restaurant owner signs in (email/password or Google), is walked through an onboarding wizard, builds a menu (categories, dishes, social links, theme) in a custom Livewire/Blade dashboard, generates a QR code, and exposes a public, mobile-first, multi-language menu page at `https://<app>/{slug}`. The product also offers an AI "menu scan" feature that extracts a menu from a photo via Google Gemini, per-restaurant visitor analytics, and a Filament admin back office.

> Note: the historical `PROJECT_PLAN.md` still describes the app under its old name "MenuX" with a broader "restaurant owner / home cook" framing and a multi-menu model. The shipped code has narrowed this to a single restaurant per user with two roles (admin, menu owner). Treat `PROJECT_PLAN.md` as aspirational/legacy; the code (and this document) is the source of truth. The boilerplate `README.md` is the stock Laravel readme and contains no project-specific information.

### 1.2 The three actors

The app is segmented into three audiences, each with its own routing namespace, middleware stack, and UI layer:

| Actor | Entry point | Auth | UI layer | Code namespace |
|-------|-------------|------|----------|----------------|
| **Public menu visitor** | `GET /{slug}` (+ `track-*` beacons) | none (rate-limited, CSRF-exempt beacons) | Blade portal views | `App\Http\Controllers\Portal`, `resources/views/portal` |
| **Restaurant owner** | `/dashboard`, `/restaurant`, `/dishes`, `/menu-scan`, `/qr-code`, … | `auth` + `EnsureUserIsMenuOwner` + onboarding/setup gates | custom Livewire 3 + Blade dashboard | `App\Http\Controllers\MenuOwner` |
| **Admin** | `/admin` (Filament panel) | `auth` + `EnsureUserIsAdmin` | Filament v4 panel | `App\Filament\Admin`, `App\Providers\Filament\AdminPanelProvider` |

### 1.7 Bootstrap: middleware & route registration (`bootstrap/app.php`)

This Laravel 12 app has no `Http/Kernel.php`; everything is configured in `bootstrap/app.php`:

- **Routing** (bootstrap/app.php:8-12): `web` routes from `routes/web.php`, console from `routes/console.php`, and a health endpoint at `/up`.
- **Trusted proxies** from the `TRUSTED_PROXIES` env (bootstrap/app.php:14).
- **Global middleware** (bootstrap/app.php:20-23):
  - `SecurityHeaders` is **prepended** so it wraps every response (sets `X-Frame-Options: DENY`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, a `frame-ancestors 'none'` CSP, and HSTS on HTTPS — see SecurityHeaders.php:20-28).
  - `BlockAbusiveIps` is **appended** (runs *after* `StartSession`) so its admin-bypass can see the authenticated user; a prepended copy would run before sessions exist.
- **CSRF exemptions** (bootstrap/app.php:29-32): `*/track-exit` and `*/track-whatsapp-order` — the analytics beacons fired via `navigator.sendBeacon` on page exit, which may run after the token expires.
- **Middleware aliases** (bootstrap/app.php:33-36): `owner.locale` → `SetMenuOwnerLocale`, `onboarding.complete` → `EnsureOnboardingComplete`.

Application middleware classes (`app/Http/Middleware/`):

| Middleware | Responsibility |
|-----------|----------------|
| `SecurityHeaders` | Adds security response headers (global, prepended) |
| `BlockAbusiveIps` | Rejects requests from blocked IPs via `AbuseGuard` (global, appended) |
| `SetMenuOwnerLocale` (`owner.locale`) | Reads `owner_locale` from session and calls `app()->setLocale()` if supported (SetMenuOwnerLocale.php:18-22) |
| `EnsureOnboardingComplete` (`onboarding.complete`) | Forces incomplete users into the onboarding wizard |
| `EnsureRestaurantSetupComplete` | Forces users without a restaurant to finish setup |
| `EnsureUserIsMenuOwner` | Gates the owner dashboard routes |
| `EnsureUserIsAdmin` | Filament panel auth middleware |

### 1.8 Service providers (`bootstrap/providers.php`)

Three app providers are registered:

- **`AppServiceProvider`** — boots named rate limiters (AppServiceProvider.php:36-67). Two low-ceiling limiters that do **not** trigger bans (`auth` 5/min, `contact` 10/min) and three high-volume limiters that feed the auto-ban on sustained abuse (`mutations` 120/min, `uploads` 20/min, `public` 120/min). Each is keyed by user id falling back to IP; on a 429 the high-volume ones call `AbuseGuard::recordViolation($ip)`.
- **`AdminPanelProvider`** (Filament) — defines the `admin` panel at `/admin` with Amber primary color, `->login()`, 10 resources (Users, Restaurants, Templates, Categories, Dishes, RestaurantSocialLinks, RestaurantStatistics, ContactMessages, Tags, BlockedIps), auto-discovered pages/widgets plus 5 explicit dashboard widgets, and `authMiddleware([Authenticate, EnsureUserIsAdmin])` on the `web` guard (AdminPanelProvider.php:33-83).
- **`TelescopeServiceProvider`** — in production filters Telescope to only record reportable exceptions, failed requests/jobs, scheduled tasks, and monitored tags; hides sensitive headers/`_token`; and exposes a `viewTelescope` gate (currently an empty allow-list, so locked down in non-local) (TelescopeServiceProvider.php:24-63).

### 1.9 Roles & app segmentation (`UserRole`)

`App\Enums\UserRole` is a string enum with exactly two cases: `Admin = 'admin'` and `MenuOwner = 'menu_owner'`. It is cast on `User::$role` (User.php:49) and drives all access control via helper predicates `isAdmin()` / `isMenuOwner()` (User.php:65-73):

- `User::canAccessPanel()` returns `isAdmin()` → only admins reach the Filament panel.
- `canImpersonate()` requires admin; `canBeImpersonated()` requires menu owner → admins can impersonate owners.
- Owner dashboard routes are wrapped in `EnsureUserIsMenuOwner`.

A second enum, `App\Enums\MenuScanStatus` (`Pending`, `Processing`, `Completed`, `Failed`), tracks the lifecycle of an async AI menu-scan job.

The `User` model also carries onboarding state (`onboarding_step`, `onboarding_completed_at`), `SoftDeletes`, a `HasOne` `restaurant` relationship, a `HasMany` `socialAccounts` relationship, and convenience gates `hasCompletedOnboarding()`, `isRestaurantSetupComplete()`, `currentOnboardingStep()` (capped at step 6).

### 1.10 Multi-language

Languages are centralized in `config/locales.php` — a single `$locales` map of 14 languages (en, ar, fr, de, es, it, pt, ru, tr, hi, zh, ja, ko, nl) with name/flag/RTL metadata; `supported`, `rtl`, and `default` are derived from it. Arabic is flagged RTL. The active locale is stored in the session key `owner_locale` (set via `/locale/{locale}` and `/owner/locale/{locale}` switch routes in routes/web.php) and applied per-request by `SetMenuOwnerLocale`. Menu content itself is translated per-locale through `spatie/laravel-translatable`.

### 1.11 Request lifecycle — how the pieces fit together

A typical request flows as follows:

1. `SecurityHeaders` (prepended) wraps the eventual response with hardening headers.
2. The Laravel `web` group runs (cookies, `StartSession`, CSRF — except the exempt `track-*` beacons).
3. `BlockAbusiveIps` (appended, post-session) rejects IPs flagged by `AbuseGuard`.
4. Route-group middleware applies per audience:
   - **Public** (`/{slug}`, `track-*`): `owner.locale` + `throttle:public`; handled by `Portal\MenuController` → portal Blade views.
   - **Owner** (`/dashboard` and the `menu-owner.*` routes): `auth` → `EnsureOnboardingComplete` → `EnsureRestaurantSetupComplete` → `owner.locale`, then `EnsureUserIsMenuOwner` + `throttle:mutations` (uploads/scan additionally `throttle:uploads`).
   - **Admin** (`/admin`): Filament's middleware stack + `Authenticate` + `EnsureUserIsAdmin`.
5. Rate-limit hits on high-volume limiters feed `AbuseGuard::recordViolation()`, which can auto-ban an IP after sustained flooding (strike threshold 20/min, see AbuseGuard.php:21); blocks are cached and skipped for trusted IPs (`SECURITY_TRUSTED_IPS`).
6. Cross-cutting services back the controllers: `Services/ThirdParty/GeminiMenuExtractorService` (AI menu scan), `Services/ThirdParty/Captcha` (reCAPTCHA v3, gracefully no-op when keys absent), `Services/MenuOwner/Entitlements` (per-plan dish/category/social-link limits, defaults in `config/entitlements.php`), `Services/MenuOwner/StatisticsQueryService` (owner analytics), `Services/Portal/DeviceDetectionService` + `ContactService`, and `Services/Global/MediaService` + `AbuseGuard`.

## 2. Domain Models & Relationships

The domain centers on the **`Restaurant`** aggregate root. A `User` (menu owner) owns exactly one restaurant; that restaurant assembles all menu content (`Category`, `Dish`, `DishVariant`, `Tag`), presentation (`Template`, `RestaurantSocialLink`), monetization (`Subscription`, `Payment`, `RestaurantFeature` over a catalog of `Feature`/`FeaturePrice`/`TemplatePrice`), and analytics (`RestaurantStatistic` raw rows + `RestaurantDailyStat` rollup + `MenuScan`). All owner-facing content is multi-language via `spatie/laravel-translatable`, and images are handled by `spatie/laravel-medialibrary`.

### 2.1 Cross-cutting design patterns

- **Translatable JSON columns** — `spatie/laravel-translatable`'s `HasTranslations` trait stores translated strings as JSON (`{"ar": "...", "en": "..."}`) in the same physical column. Each model declares a public `array $translatable`. The migration `database/migrations/2026_06_12_134515_convert_content_columns_to_translatable_json.php` converted the legacy plain columns to JSON, backfilling restaurant-owned content under the restaurant's `default_locale` and platform-owned content (tags, restaurant types, templates) under `en`.
- **Media Library** — models implementing `Spatie\MediaLibrary\HasMedia` register collections in `registerMediaCollections()`. Files default to private visibility (Filament v4 default).
- **Entitlement cache invalidation** — `Restaurant`, `Subscription`, and `RestaurantFeature` flush the per-restaurant entitlements cache on save/delete (and `Restaurant` on `template_id` change) via `App\Services\MenuOwner\Entitlements::flush()`. See `app/Models/Subscription.php:46-52`, `app/Models/RestaurantFeature.php:32-38`, `app/Models/Restaurant.php:66-70`.
- **Enums** — `App\Enums\UserRole` (`Admin = 'admin'`, `MenuOwner = 'menu_owner'`) and `App\Enums\MenuScanStatus` (`Pending`, `Processing`, `Completed`, `Failed`) are backed string enums cast on their models.

### 2.3 The aggregate root: Restaurant

- **Table:** `restaurants` (one row per user — `unique('user_id')` added in `2026_06_12_134511_phase0_restaurants_hygiene.php`).
- **Translatable:** `name`, `description`, `address`.
- **Casts:** `is_active` → boolean; `template_settings`, `qr_settings` → array (JSON).
- **Fillable highlights:** `user_id`, `slug`, `country_code`, `phone`, `template_id`, `template_settings`, `qr_settings`, `google_maps_url`, `currency`, `default_locale` (owner content base language, `ar`/`en`, default `ar`), `timezone` (per-restaurant, default `UTC`), `restaurant_type_id`.
- **Model events (`booted()`):** on `creating`, auto-generates a unique `slug` from the name (falling back to `menu`, then `-2`, `-3`… on collision, app/Models/Restaurant.php:53-64); on `saved`, flushes entitlements when `template_id` changed.
- **Media collections:** `logo` (single file, jpeg/png/webp) and `cover_image` (single file, jpeg/png/webp). Note: an earlier migration re-pointed `logo`/`cover_image` media from `User` to `Restaurant`.
- **Entitlement helpers (delegating to the cached `Entitlements` service):**
  - `entitlements()` returns `Entitlements::for($this)`.
  - Accessors `dish_limit`, `category_limit`, `social_link_limit` resolve numeric limits via `entitlements()->limit(...)` (app/Models/Restaurant.php:171-184).
  - `hasReachedDishLimit()`, `getRemainingDishSlots()` and the category/social-link equivalents gate content creation against those limits.
- **Publishing gate:** `menuIsPubliclyAvailable()` (app/Models/Restaurant.php:162) returns true if there is no template or the template `isFree()`; otherwise requires an `activeTemplateSubscription()`. `activeTemplateSubscription()` (app/Models/Restaurant.php:143) finds the latest `active()` subscription whose `subscribable` is the assigned `Template`. Expiry takes the menu offline without deleting data.
- **Template settings:** `resolvedTemplateSettings()` merges the template's `default_settings` with the owner's stored `template_settings` overrides (via `Template::resolveSettings()`).
- **Analytics aggregations (read off `statistics()` → `menu_sessions`):** `getTotalViews()`, `getUniqueVisitors()` (distinct `session_id`), `getAverageTimeSpent()`, `getTotalTimeSpent()`, `getQrScanCount($period)` and `getWhatsAppOrderCount($period)` — the last two accept `today`/`week`/`month`/`all` and switch via a `match` over date filters (app/Models/Restaurant.php:236-265).

### 2.4 Identity & access: User, SocialAccount

**User** (`users`, extends `Authenticatable`, implements `FilamentUser`):
- **Traits:** `HasFactory`, `Impersonate` (Lab404 — admins impersonate menu owners), `Notifiable`, `SoftDeletes`.
- **Casts:** `email_verified_at`/`onboarding_completed_at` → datetime, `password` → hashed, `role` → `UserRole` enum, `onboarding_step` → integer.
- **Role helpers:** `isAdmin()`, `isMenuOwner()`. `canAccessPanel()` restricts the Filament panel to admins; `canImpersonate()`/`canBeImpersonated()` enforce admin→menu-owner impersonation only.
- **Onboarding/profile:** `isRestaurantSetupComplete()` / `isProfileComplete()` check the `restaurant` relation exists; `hasCompletedOnboarding()` checks `onboarding_completed_at`; `currentOnboardingStep()` returns `min(onboarding_step + 1, 6)`. Password is nullable (social-only signups).

**SocialAccount** (`social_accounts`): OAuth identities per user. **Encrypted casts** on `access_token` and `refresh_token` (`'encrypted'`), `token_expires_at` → datetime. No `HasFactory`/`HasTranslations`. A migration encrypted previously-stored tokens; `avatar` is a `text` column.

### 2.5 Menu content: Category, Dish, DishVariant, Tag

| Model | Table | Translatable | Notable casts | Media |
| --- | --- | --- | --- | --- |
| Category | `categories` | `name`, `description` | `restaurant_id`, `display_order` → int | `image` (single file) |
| Dish | `dishes` | `name`, `ingredients` | `price` → decimal:2, `is_available` → bool, `display_order`/ids → int | `images` (multi, jpeg/png/webp) |
| DishVariant | `dish_variants` | `name` | `price` → decimal:2, `is_available` → bool, `display_order` → int | none |
| Tag | `tags` | `name` | none (no casts method) | none |

- `Category`/`Dish` order children by `display_order` in their `hasMany` relations.
- `Dish` connects to `Tag` through the `dish_tag` pivot and owns ordered `DishVariant` rows (size/portion variants with their own price and availability).
- `Tag` carries a non-translatable `slug` and a `category` grouping column and is the only menu-content model that shares across `Restaurant`, `Template`, and `Dish` (many-to-many in all three). It uses `HasTranslations` but not `HasFactory`.

### 2.6 Presentation: Template, TemplatePrice, RestaurantType, RestaurantSocialLink

**Template** (`templates`, implements `HasMedia`): the "plan" a restaurant runs on.
- **Translatable:** `name`, `description`. **Casts:** `capabilities`, `default_settings` → array; `is_active` → bool; `sort_order` → int. Fillable includes `tier` (`free`/`paid`).
- **Tier/capability logic:** `isFree()` (true unless `tier === 'paid'`), `isPaid()`, `capability($key, $default = true)` reads a boolean flag from the `capabilities` JSON, `defaultSettings()`, and `resolveSettings($stored)` (merges defaults with non-null stored overrides — app/Models/Template.php:89-95).
- **Media:** `thumbnail` (single file).
- Bundles `Feature`s through `template_feature` (`withPivot('value')`), has `TemplatePrice`s, and is a polymorphic `subscribable` for `Subscription`.

**TemplatePrice** (`template_prices`): `period` (e.g. monthly/yearly), `price` (decimal:2), `currency` (char(3), default `USD`), `is_active` (bool). belongsTo `Template`.

**RestaurantType** (`restaurant_types`): translatable `name`, plus `slug` and `icon`; `hasMany` restaurants. Uses `HasTranslations` only.

**RestaurantSocialLink** (`restaurant_social_links`): `platform` + `url` per restaurant (display_order/is_active were removed in a later migration). `restaurant_id` cast to int.

### 2.7 Monetization & entitlements

This cluster implements the documented model: **templates are the plans, features are the sellable units (booleans or limits), bundled via `template_feature` or sold as add-ons; subscriptions/payments support manual billing first** (`2026_06_12_134516_create_entitlement_tables.php`).

| Model | Table | Purpose / key fields |
| --- | --- | --- |
| Feature | `features` | Sellable unit. Translatable `name`/`description`; `slug` (unique), `kind` (`boolean`\|`limit`), `is_addon`, `is_active`. `isLimit()`/`isLimit` distinguishes numeric limits from on/off flags. |
| FeaturePrice | `feature_prices` | Price points per feature: `period`, `price` (decimal:2), `currency` (char(3) `USD`), `is_active`. No `HasFactory`. |
| RestaurantFeature | `restaurant_features` | A granted/purchased feature on a restaurant: `value`, `source` (default `purchase`), time-boxed via `starts_at`/`ends_at` (datetime), optional `payment_id`. Flushes entitlement cache on save/delete. |
| Subscription | `subscriptions` | Recurring entitlement. Polymorphic `subscribable` (Template or Feature), `period`, `price`, `currency`, `status`, `trial_ends_at`/`starts_at`/`current_period_end`/`canceled_at` (datetime), `provider`/`provider_ref`, `meta` (array). |
| Payment | `payments` | Polymorphic `payable` (Subscription) on a restaurant: `amount` (decimal:2), `currency`, `status`, `provider` (default `manual`), `provider_ref`, `paid_at`, `meta` (array). |

**Subscription entitlement semantics** (`app/Models/Subscription.php`):
- `scopeActive()` (line 73) selects subscriptions that are `trialing`/`active` with `current_period_end` in the future, **or** `past_due` within the configured grace window (`config('entitlements.grace_days', 7)`, default 7 days).
- `isEntitling()` (line 88) is the per-instance equivalent using a `match` on `status`.

**Entitlement resolution** (`app/Services/MenuOwner/Entitlements.php`) — the service that ties these models together:
- `Entitlements::for($restaurant)` is reachable from `Restaurant::entitlements()`. Results are cached per restaurant (`entitlements:{id}`, TTL `config('entitlements.cache_ttl', 300)`).
- `resolve()` layers three sources: (1) the assigned **template's** bundled features — but only if the template `isFree()` or has an `activeTemplateSubscription()`; (2) active **feature add-on subscriptions** (`subscribable_type = Feature`); (3) valid **`restaurant_features`** grants within their `starts_at`/`ends_at` window.
- `merge()` (line 113) combines overlapping entitlements: **limits take MAX, booleans take OR**.
- `limit($slug)` falls back to `config('entitlements.defaults.*')` (`dish_limit` 40, `category_limit` 10, `social_link_limit` 4) when no entitlement is present; booleans default to `false`.

### 2.8 Analytics: RestaurantStatistic, RestaurantDailyStat, MenuScan

**RestaurantStatistic** (`$table = 'menu_sessions'`): one raw row per visit. The table was renamed from `restaurant_statistics` to `menu_sessions` (`2026_06_12_134520_...`) — raw rows are pruned after ~6 months. Fields: `session_id`, `device_type`, `browser`, `os`, `viewed_at`, `time_spent`, `page_views`, `via_qr` (bool), `whatsapp_orders` (int). These rows feed the `Restaurant` analytics aggregations in §2.3.

**RestaurantDailyStat** (`$table = 'restaurant_stats_daily'`): the nightly rollup dashboards read. One row per `(restaurant_id, date)` (unique): `visits`, `unique_sessions`, `qr_visits`, `whatsapp_orders`, `avg_time_spent`; `date` cast to date, the rest to int.

**MenuScan** (`menu_scans`, implements `HasMedia`): a QR/menu-image OCR import job. `status` cast to `MenuScanStatus` enum (default attribute `pending`), `result` cast to array. Media collection `scan` (single file, stored on the **`local`** disk, jpeg/png/webp). State helpers `isPending()`/`isCompleted()`/`isFailed()` and mutators `markProcessing()`, `markCompleted(array $result)`, `markFailed(string $error)` drive the async scan lifecycle.

### 2.9 Operational / standalone models

- **ContactMessage** (`contact_messages`): public contact-form submissions. Fillable `name`, `email`, `message`, `ip_address` (indexed for abuse correlation). No relationships, casts, or traits.
- **BlockedIp** (`blocked_ips`): IP blocklist for abuse control. Fillable `ip`, `reason`, `expires_at` (datetime). `scopeActive()` (app/Models/BlockedIp.php:33) matches rows with a null `expires_at` (permanent) or a future expiry.

### 2.10 Note: ApiResponse trait

`app/Traits/ApiResponse.php` is a controller helper, **not** a model concern. It standardizes the JSON envelope (`success`/`error`/`created`/`noContent`) with a `{ success, message?, data? }` / `{ success, errors? }` shape mirroring Laravel's validation error format. It does not participate in the domain model layer.

## 3. Database Schema & Migration History

The database is MySQL (database name `menux`) on the default connection, with Telescope able to use its own connection. The schema reached its current shape through ~60 migrations that fall into four eras: the **original menus/dishes prototype** (Dec 2025 – Jan 2026), the **restaurants & templates restructure** (May 4 2026), a **feature-build-out** (May–Jun 2026), and the **"phase0" hardening batch** (Jun 12 2026) that introduced i18n, entitlements, variants, and analytics rollups. The committed ERD in `docs/database-erd.md` documents a 23-table snapshot taken on 2026-06-09 — it is therefore **pre-phase0** and does not reflect the translatable-JSON conversion, `dish_variants`, the entitlement tables, `menu_sessions`/`restaurant_stats_daily`, or `telescope_entries`. This section reflects the live post-phase0 schema.

### 3.1 Schema Evolution Narrative

#### Era 1 — Original "menus" prototype (2025-12-20 → 2026-01-03)
The app began with a `menus`-centric model. `users` (`0001_01_01_000000_create_users_table.php`) was the standard Breeze table; `2025_12_20_111233_add_role_to_users_table.php` added `role` (default `menu_owner`) plus restaurant-identity columns on the user itself (`restaurant_name`, `phone`, `address`). The menu domain was:

- `menus` — `user_id` FK, `name`, `slug` (unique), `menu_style` enum (`restaurant`/`home`), `is_active`, `dish_limit` (default 20).
- `categories` and `dishes` both hung off `menu_id`. `dishes` was rich: `description`, `price`, `ingredients` (text), `allergens` (json), `prep_time`, `serving_size`.
- A generic key/value settings system: `settings` (global definitions) + `menu_settings` (per-menu values, unique on `menu_id`+`setting_id`).
- `menu_social_links` and a heavyweight `menu_statistics` table (ip, user_agent, referrer, country, device, `left_at`, `interactions` json).
- `media` (Spatie medialibrary) was created first (`2025_12_20_111213`).

Three January migrations added `category_limit` (default 10) and `social_link_limit` (default 4) to `menus`, and stripped `display_order`/`is_active` from `menu_social_links`.

A May-4 cleanup batch (timestamps `2026_05_04_1212xx`) trimmed the prototype just before the big restructure: dropped `country`/`interactions`, then `ip_address`/`user_agent`/`referrer`/`left_at` from `menu_statistics`; dropped `allergens`/`prep_time`/`serving_size` from `dishes`; dropped `menu_style` from `menus`.

#### Era 2 — Restructure to restaurants & templates (2026-05-04)
`2026_05_04_202309_restructure_to_restaurants_and_templates.php` is the pivotal migration. In one transaction it:
1. Creates `templates` (`name`, `slug`, `description`, `fields` json, `is_active`).
2. Creates `restaurants` (the new aggregate root) with `user_id`, `name`, `slug`, `country_code`, `phone`, `is_active`, the three `*_limit` columns, and `template_id` (nullable, `nullOnDelete`) + `template_settings` json.
3. **Data migration**: copies every `menus` row into a `restaurants` row, preferring the user's `restaurant_name`/`phone` when present (`:42-58`).
4. **Re-points media**: `logo`/`cover_image` media rows owned by `App\Models\User` are moved onto the corresponding `App\Models\Restaurant` (`:60-74`).
5. Creates `restaurant_social_links` and `restaurant_statistics`, backfilling both from the legacy tables.
6. Re-parents `categories` and `dishes` from `menu_id` to `restaurant_id` (add nullable column, backfill, drop old FK/column, then `MODIFY ... NOT NULL` on MySQL — `:137-183`).
7. Drops the legacy tables (`menu_social_links`, `menu_statistics`, `menu_settings`, `settings`, `menus`) and the now-redundant user columns (`restaurant_name`, `phone`, `address`).

The week that followed enriched `restaurants`: `qr_settings` json + a `via_qr` flag on statistics (`..._160730`); `address`, `google_maps_url`, `currency` (default USD); `preferred_language` (default `en`); and `whatsapp_orders` on statistics. `dishes.ingredients` flipped json → text → (later) json again as the UI evolved (`..._100433`, `..._133218`).

#### Era 3 — Accounts, AI scans, taxonomy, capabilities (2026-05-08 → 2026-05-13)
- **Auth/onboarding**: `users.password` made nullable (Google-only accounts), `onboarding_step`/`onboarding_completed_at` added (existing users backfilled to step 3 / completed — `..._145548`), soft deletes added, and `social_accounts` created for OAuth (`provider`, `provider_user_id` unique-with-provider, `access_token`/`refresh_token`, `token_expires_at`, `avatar` — later widened to text).
- **AI menu scanning**: `menu_scans` (`restaurant_id`, `status` defaulting to `MenuScanStatus::Pending`, `image_path`, `result` json, `error`), indexed on `restaurant_id`+`status`.
- **Taxonomy**: `restaurant_types` (with `icon`), `tags` (enum `category`: cuisine/dietary/vibe/style), `restaurant_type_id` added to `restaurants`, and the pivots `restaurant_tag` / `template_tag` (composite PKs).
- **Templates capabilities**: `..._194234` added 20 `has_*` boolean capability flags + `default_direction`/`allows_direction_change` directly as columns on `templates`.
- **Dish tags churn**: a `dish_tag` pivot was created (`..._195805`) then immediately replaced by a free-form `dishes.tags` json column (`..._200602`) — this is later reversed in phase0.
- **Contact form**: `contact_messages` (`email`, `message`, `ip_address`); `name` added the next day.

#### Era 4 — Security hardening (2026-06-01 → 2026-06-07)
`2026_06_01_100334_encrypt_existing_social_account_tokens.php` is a **data-only, idempotent** migration: it chunks through `social_accounts` and `Crypt::encryptString`s any plaintext `access_token`/`refresh_token` (detecting already-encrypted values via a try/catch decrypt), so the rows decrypt cleanly under the model's new `encrypted` casts. A sibling migration added explicit `restaurant_id` indexes to `categories`/`dishes` (some drivers don't auto-index FKs). `blocked_ips` (`ip` indexed, `reason`, `expires_at` nullable = permanent) was added for abuse mitigation, and `contact_messages.ip_address` was indexed for throttling.

#### Era 5 — "phase0" hardening batch (2026-06-12) — the current shape
Nine migrations dated `2026_06_12_1345xx` reshape the domain for i18n and monetization:

- **`..._134511_phase0_restaurants_hygiene`**: enforces **one restaurant per user** (`unique('user_id')`); adds per-restaurant `timezone` (default UTC); tightens `country_code`→`char(2)` and `currency`→`char(3)`; renames `preferred_language`→`default_locale` and constrains it to `ar`/`en` (`char(2)`, default `ar`).
- **`..._134512_create_dish_variants`**: `dish_variants` (`dish_id`, translatable `name` json, `price`, `is_available`, `display_order`) — multiple priced sizes per dish.
- **`..._134514_create_dish_tag`**: recreates the curated `dish_tag` pivot, best-effort matching the old free-form `dishes.tags` strings against `tags.slug` (unmatched strings dropped by design), then drops `dishes.tags`.
- **`..._134515_convert_content_columns_to_translatable_json`**: the i18n keystone. Converts content columns to `spatie/laravel-translatable` JSON (`{"ar":…,"en":…}`) by adding a `*_i18n` column, backfilling in PHP chunks, dropping the original, and renaming. Restaurant-owned content (`restaurants.name/description/address`, `categories.name/description`, `dishes.name/ingredients`) is backfilled under the restaurant's `default_locale`; platform-owned content (`tags.name`, `restaurant_types.name`, `templates.name/description`) under `en`.
- **`..._134516_create_entitlement_tables`**: the monetization engine — `features`, `feature_prices`, `template_prices`, `template_feature` (pivot with `value`), `subscriptions` (polymorphic `subscribable`), `payments` (polymorphic `payable`, default provider `manual`), and `restaurant_features` (granted entitlements with `source`, validity window, optional `payment_id`). See §3.2.
- **`..._134518_drop_limit_columns_from_restaurants`**: removes `dish_limit`/`category_limit`/`social_link_limit` — limits now come from limit-kind features on the (free) template, overridable per restaurant.
- **`..._134519_add_capabilities_and_tier_to_templates`**: collapses the 20 `has_*` flags + direction columns into a single `capabilities` json, converts the old `fields` schema into a precomputed `default_settings` map, and adds `tier` (all existing templates → `free` so no live menu goes dark) + `sort_order`. Drops the flag/direction/`fields` columns.
- **`..._134520_rename_statistics_to_menu_sessions_and_create_daily_rollup`**: renames `restaurant_statistics`→`menu_sessions` (raw rows, pruned after ~6 months) and creates `restaurant_stats_daily` (nightly rollup dashboards read; unique on `restaurant_id`+`date`).
- **`..._134521_move_menu_scan_images_to_media`**: drops `menu_scans.image_path`; scan uploads now use the medialibrary (`scan` collection on `MenuScan`).

Finally, `2026_06_14_105141_create_telescope_entries_table.php` adds Laravel Telescope's `telescope_entries`, `telescope_entries_tags`, `telescope_monitoring` (on the Telescope storage connection from config).

### 3.3 Cross-cutting patterns

- **Idempotent / driver-portable migrations**: data conversions (token encryption, ingredients json↔text, i18n backfill, dish-tag matching) run in **PHP chunks** (`chunkById`) rather than SQL so they work on both MySQL and the SQLite test database. Index migrations guard with `Schema::hasIndex(...)`, and `2026_05_04_122005` explicitly drops the `ip_address` index before the column because SQLite errors on a dangling index during a fresh test migration.
- **Translatable content** lives as JSON `{"ar":…,"en":…}` (Spatie translatable) on restaurants, categories, dishes, dish_variants, tags, restaurant_types, templates.
- **Capabilities/limits are data, not columns**: template behavior moved from 20 `has_*` boolean columns into `templates.capabilities` json, and quota limits moved from `restaurants.*_limit` columns into the feature/entitlement tables.
- **Polymorphic ownership**: `media` (Spatie), plus `subscriptions.subscribable` and `payments.payable` use morphs so a charge/subscription can target either a template (plan) or an individual feature (add-on).

## 4. Routing, Authentication & Onboarding

This section maps every HTTP entry point of qayema, the credential and Google-OAuth login flows, the six-step onboarding wizard, the chain of middleware that gates the menu-owner dashboard until both onboarding and restaurant setup are finished, and how a menu owner's UI language is resolved on every request.

### 4.1 Route files & how they are loaded

`bootstrap/app.php` registers only `routes/web.php` as the web route file (plus `routes/console.php` for the scheduler and `/up` as the health check). `routes/web.php:38` then `require`s `routes/auth.php`, so all authentication and onboarding routes are merged into the same web group. There is no `routes/api.php` and no dedicated admin route file — the admin surface is a Filament panel mounted at `/admin` by `app/Providers/Filament/AdminPanelProvider.php` (see 4.6).

Named middleware aliases are registered in `bootstrap/app.php:33-36`:

| Alias | Class |
| --- | --- |
| `owner.locale` | `App\Http\Middleware\SetMenuOwnerLocale` |
| `onboarding.complete` | `App\Http\Middleware\EnsureOnboardingComplete` |

The other gate middlewares (`EnsureRestaurantSetupComplete`, `EnsureUserIsMenuOwner`, `EnsureUserIsAdmin`) are referenced by their fully-qualified class name directly in the route groups rather than by alias. Two global middlewares are also wired in `bootstrap/app.php:20-23`: `SecurityHeaders` is *prepended* to the web stack, and `BlockAbusiveIps` is *appended* (so it runs after `StartSession` and can see the authenticated user for its admin bypass). The two analytics tracking endpoints are CSRF-exempt (`bootstrap/app.php:29-32`, patterns `*/track-exit` and `*/track-whatsapp-order`) because they are fired via `navigator.sendBeacon` on page exit when the session token may already be gone.

Throttle middleware names (`throttle:auth`, `throttle:mutations`, etc.) resolve to named limiters defined in `app/Providers/AppServiceProvider.php:36-49`. Each limiter is keyed by `user id` falling back to IP (`AppServiceProvider.php:58`):

| Limiter | Ceiling | Feeds abuse auto-ban? |
| --- | --- | --- |
| `auth` | 5/min | No (`autoBan: false`) |
| `contact` | 10/min | No (`autoBan: false`) |
| `mutations` | 120/min | Yes |
| `uploads` | 20/min | Yes |
| `public` | 120/min | Yes |

Low-ceiling limiters (`auth`, `contact`) deliberately do not escalate to an IP ban so a shared office NAT cannot be locked out by one busy user (`AppServiceProvider.php:38-42`).

### 4.5 Authentication flows

#### Credential login (`AuthenticatedSessionController` + `LoginRequest`)

`store()` delegates validation/auth to `LoginRequest`, regenerates the session, and redirects to the intended URL or `dashboard` (`AuthenticatedSessionController.php:25-32`). `LoginRequest` (`app/Http/Requests/Auth/LoginRequest.php`) does the heavy lifting:

- Rules: `email` required/email, `password` required, plus a custom `g-recaptcha-response` validated by `App\Rules\ValidCaptcha` (`LoginRequest.php:30-37`).
- `authenticate()` (`LoginRequest.php:44-67`) first calls `ensureIsNotRateLimited()` — 5 attempts per `email|ip` key, firing a `Lockout` event and a localized `auth.throttle` message on the 6th (`LoginRequest.php:74-90`). It then **blocks password login for Google-only accounts**: if a matching user exists with `password === null`, it throws a localized `auth.google_only` validation error (`LoginRequest.php:49-56`). Otherwise it calls `Auth::attempt()` with the `remember` flag, throwing `auth.failed` on failure and clearing the rate-limit key on success.
- Throttle key: transliterated lowercase email + IP (`LoginRequest.php:95-98`).

`destroy()` logs out the web guard, invalidates the session, regenerates the CSRF token, and redirects to `login` (`AuthenticatedSessionController.php:37-46`).

#### Google OAuth (`GoogleController` via Laravel Socialite)

Configured by `config/services.php:43-47` (`google.client_id`, `google.client_secret`, `google.redirect`). `redirect()` sends the user to Google (`GoogleController.php:16-19`). `callback()` (`GoogleController.php:21-75`) implements a three-branch account resolution:

1. **Wrapped failure** — any `Throwable` from `Socialite::driver('google')->user()` redirects back to `login` with a flashed `error` (`GoogleController.php:23-28`).
2. **Existing social account** — looked up by `provider='google'` + `provider_user_id`. Its `access_token` / `refresh_token` / `token_expires_at` are refreshed, the user is logged in with `remember: true`, and they land on the intended URL or `dashboard` (`GoogleController.php:30-46`).
3. **Existing user by email** — if a `User` already exists for the Google email, it is force-verified (`email_verified_at`), a `SocialAccount` is attached, and the user is logged in and sent to `dashboard` (`GoogleController.php:48-59`).
4. **Brand-new user** — created with `password = null`, `role = UserRole::MenuOwner`, `onboarding_step = 0`, force-verified, given a `SocialAccount`, logged in, and redirected to **`onboarding`** (`GoogleController.php:61-74`).

`attachSocialAccount()` stores the provider tokens and avatar (`GoogleController.php:77-89`). Because new Google users have a null password, they are permanently steered to OAuth on any later credential-login attempt (the `auth.google_only` guard above).

#### Social account storage & token encryption (`SocialAccount`)

`app/Models/SocialAccount.php` casts `access_token` and `refresh_token` as `'encrypted'` (`SocialAccount.php:23-24`) so OAuth tokens are encrypted at rest, with `token_expires_at` cast to `datetime`. Schema (`database/migrations/2026_05_08_145549_create_social_accounts_table.php`): `user_id` FK with `cascadeOnDelete`, `provider`, `provider_user_id`, nullable `text` `access_token` / `refresh_token` (text to hold the larger ciphertext), `token_expires_at`, `avatar`, and a **unique composite index on `(provider, provider_user_id)`** — the same key `GoogleController` queries on. A later migration widened `avatar` to `text`. `User::socialAccounts()` is the inverse `hasMany` (`User.php:60-63`).

#### Password change (`PasswordController`)

`update()` (`app/Http/Controllers/Auth/PasswordController.php:13-31`) requires `current_password` only when the user already has one (Google-only users can set their first password without it), validates the new password against `Password::defaults()` with confirmation under the `updatePassword` error bag, hashes it, and flashes a `password-updated` status.

### 4.6 Authorization & the onboarding gate chain

Roles come from `App\Enums\UserRole` (`Admin = 'admin'`, `MenuOwner = 'menu_owner'`), cast on `User::$role`. Helper predicates live on `User`: `isAdmin()`, `isMenuOwner()`, `isRestaurantSetupComplete()` (true when a `restaurant` exists), `hasCompletedOnboarding()` (true when `onboarding_completed_at` is set), and `currentOnboardingStep()` = `min(onboarding_step + 1, 6)` (`User.php:65-93`).

The dashboard group stacks the gates in this order (`routes/web.php:57`):

| Middleware | Behavior | Admin bypass |
| --- | --- | --- |
| `EnsureOnboardingComplete` (`onboarding.complete`) | If a non-admin user has not completed onboarding, redirect to `onboarding` — unless the current route is `onboarding`, `onboarding.advance`, `logout`, or `menu-owner.temp-upload` (so the wizard and its image upload still work). | Yes (`!$user->isAdmin()`) |
| `EnsureRestaurantSetupComplete` | If the user has no `restaurant`, redirect to `menu-owner.restaurant.index` — unless on `menu-owner.restaurant.*`, `profile.*`, or `logout`. | Yes (returns `$next` early for admins) |
| `EnsureUserIsMenuOwner` (inner group) | `abort(403)` unless the user is a menu owner **or** an admin. | Allowed |
| `EnsureUserIsAdmin` | `abort(403)` unless authenticated and `isAdmin()`. Used only by the Filament admin panel, not the web routes. | n/a |

Net effect: a freshly authenticated owner who has not finished the wizard is bounced to `/onboarding`; once `onboarding_completed_at` is set but no restaurant row exists yet, they are bounced to `/restaurant`; only then can they reach the rest of the dashboard. Admins skip both content gates entirely.

**Admin surface** is the Filament panel at `/admin` (`AdminPanelProvider.php:37-39`), guarded by `authMiddleware([Authenticate::class, EnsureUserIsAdmin::class])` (`AdminPanelProvider.php:78-81`) on the `web` guard. Panel access is additionally gated by `User::canAccessPanel()` which returns `isAdmin()` (`User.php:95-98`). Impersonation hooks (`canImpersonate()` for admins, `canBeImpersonated()` for menu owners) tie the `lab404/laravel-impersonate` integration to roles (`User.php:20-28`).

### 4.7 Six-step onboarding wizard (`OnboardingController`)

`OnboardingController` defines `TOTAL_STEPS = 6` (`OnboardingController.php:19`). `show()` redirects completed users straight to `dashboard`; otherwise it renders `auth.onboarding` with the resolved `currentOnboardingStep()`, all `Tag`s grouped by category, active `Template`s, and the user's restaurant (if any) (`OnboardingController.php:21-36`).

The wizard is AJAX-driven through `advance()` (`OnboardingController.php:59-224`). It converts the 1-based `_step` from the frontend to a 0-based index and clamps it to `0..5` so rapid "next" clicks can't run a handler past the last step (`OnboardingController.php:67-68`). If any step after the first runs without a restaurant existing, it short-circuits back to step 1 (`OnboardingController.php:71-73`). Per step:

| 0-based case | Visual step | Validates | Effect |
| --- | --- | --- | --- |
| 0 | Step 1 | `name` (2–255), server-normalized `slug` (regex `^[a-z0-9][a-z0-9-]*[a-z0-9]$`, unique on `restaurants.slug` ignoring own id), optional `default_locale` in `ar,en` | Creates or updates the `Restaurant` (name stored as a translatable `{locale: name}` map, default locale `ar`) |
| 1 | Step 2 | `country_code`, `phone` (custom regex, ≥6 digits), `currency` in `config('currencies')` keys | Updates restaurant contact + currency |
| 2 | Step 3 | optional `logo_key` / `cover_image_key` (UUID-shaped temp keys) | Promotes temp uploads via `MediaService::tempPath()` into the `logo` / `cover_image` media collections; logo stays optional so the wizard remains completable |
| 3 | Step 4 | `tag_ids[]` (≤30, exist) | Detaches all `cuisine`/`dietary` tags then re-attaches the allowed subset |
| 4 | Step 5 | `tag_ids[]` (≤30, exist) | Detaches all `vibe`/`style` tags then re-attaches the allowed subset |
| 5 | Step 6 | `template_id` (exists & `is_active=1`) | Sets `template_id` + `template_settings` (from `Template::defaultSettings()`), sets `onboarding_step=6` and `onboarding_completed_at=now()`, sends the `WelcomeRestaurantOwner` mail, and returns `{completed:true, redirect: route('dashboard')}` |

For non-final steps, the saved pointer only ever moves forward — `onboarding_step = max($dbStep, $next)` (`OnboardingController.php:219-220`) — and the response returns the next visual step capped at 6. `checkSlug()` (`OnboardingController.php:38-57`) backs the live slug-availability check: it slugifies the input, rejects anything under 2 chars, and reports availability excluding the user's own restaurant id, swallowing any error as "unavailable".

### 4.8 Locale handling for menu owners (`SetMenuOwnerLocale`)

Aliased as `owner.locale`, this middleware reads `owner_locale` from the session (default `config('locales.default')` = `en`) and, if it is in `config('locales.supported')`, calls `app()->setLocale()` (`SetMenuOwnerLocale.php:17-23`). It is applied to virtually every owner-facing and public route so portal pages, the login screen, the onboarding wizard, and the dashboard all render in the chosen language. The two `locale.switch` / `owner.locale.switch` routes write the session value (guest-side before login, owner-side after) and redirect back, so a language picked on the marketing site survives the login boundary. Supported languages, default, and RTL set are all derived from `config/locales.php` (14 locales; `ar` is the lone RTL); adding a language there plus its `lang/{code}/auth.php` and `lang/{code}/menu_owner.php` files propagates everywhere automatically.

## 5. Menu Owner Dashboard (Controllers & Requests)

The Menu Owner Dashboard is the authenticated, server-rendered (Blade + Alpine) area where a restaurant owner builds and manages their digital menu. All of its controllers live in `app/Http/Controllers/MenuOwner/` and all of their write-paths are validated by Form Requests in `app/Http/Requests/`. The dashboard is the non-Filament half of the app (Filament is the admin panel); these controllers serve traditional `View`/`RedirectResponse` pages plus a handful of JSON/SVG endpoints consumed by Alpine on the page.

### 5.1 Routing & middleware envelope

Every dashboard route is defined in `routes/web.php`. The core menu-management routes sit inside a nested middleware stack (`routes/web.php:57-115`):

```
auth → EnsureOnboardingComplete → EnsureRestaurantSetupComplete → owner.locale   (outer group, includes /dashboard)
  └─ EnsureUserIsMenuOwner → throttle:mutations                                   (inner group, all CRUD controllers)
```

Two routes inside that inner group add a stricter limiter for file work: `POST /temp-upload`, `POST /menu-scan`, and `POST /menu-scan/{id}/import` each layer `throttle:uploads` on top (`routes/web.php:65,98,100`). Both `throttle:mutations` and `throttle:uploads` are custom named limiters registered in `app/Providers/AppServiceProvider.php:56` (per-IP, with auto-ban via `AbuseGuard`). Profile routes are a separate group outside the menu-owner stack — `auth + owner.locale + throttle:mutations` (`routes/web.php:118-124`) — so the profile page works even before restaurant setup is complete.

A near-universal pattern across these controllers: `$restaurant = $request->user()->restaurant;` (a `User hasOne Restaurant`). Read screens tolerate a missing restaurant by returning an empty `collect()`; write screens redirect to `menu-owner.restaurant.index` with the `setup_first` flash error.

### 5.3 Shared services used

Three services are injected/called throughout:

- **`App\Services\Global\MediaService`** (`app/Services/Global/MediaService.php`) — the deferred-upload + media-promotion pipeline. `storeTempUpload()` optimizes an upload (Intervention Image), writes it to `storage/app/temp/{userId}/{uuid}.jpg`, and returns a UUID `key` plus human-readable size savings. `sync($model, $key, $delete, $collection, $mediaName)` later promotes that temp file into a Spatie Media Library collection on save (or clears the collection when `delete` is set). Injected into Category, Dish, Restaurant, and TempUpload controllers.
- **`App\Services\MenuOwner\Entitlements`** (`app/Services/MenuOwner/Entitlements.php`) — resolves a restaurant's effective feature flags and numeric limits (template bundle ∪ feature add-on subscriptions ∪ time-boxed `restaurant_features` grants; booleans merge with OR, limits with MAX), cached 5 min per restaurant. Controllers never call it directly — they go through `Restaurant` accessors (`getDishLimitAttribute`, `getCategoryLimitAttribute`, `getSocialLinkLimitAttribute` at `app/Models/Restaurant.php:171-184`) and the `hasReached*Limit()` / `getRemaining*Slots()` helpers (`Restaurant.php:186-214`), which call `entitlements()->limit('dish_limit')` etc.
- **`App\Services\MenuOwner\StatisticsQueryService`** (`app/Services/MenuOwner/StatisticsQueryService.php`) — builds the entire analytics payload; used only by `StatisticController`.

### 5.4 Restaurant profile (`RestaurantController`)

`index()` renders `dashboard/restaurant/index.blade.php` with the (possibly null) restaurant. `storeOrUpdate()` (`RestaurantController.php:25-68`) is the single create-or-update action:

1. Pulls validated data via `$request->safe()->except([... media keys ...])`.
2. `is_active` is derived from presence of the checkbox (`$request->has('is_active')`).
3. Slug handling: empty slug falls back to `Str::slug($name)`, then a uniqueness loop appends `-1`, `-2`, … against all other restaurants (`RestaurantController.php:37-43`).
4. **Translatable write strategy** (used identically in Category/Dish): for `name`, `description`, `address`, a submitted plain string is merged into the restaurant's base-language slot (`default_locale`, falling back to `ar`) while preserving any translation already stored for the other locale (`RestaurantController.php:47-53`).
5. Updates or creates the row (`user_id` set on create), then `MediaService::sync()` is called twice — once for the `logo` collection, once for `cover_image`.

**`RestaurantRequest`** (`app/Http/Requests/RestaurantRequest.php`): `authorize()` allows menu owners and admins. `prepareForValidation()` pre-slugifies any submitted slug so spaces/uppercase never reach the regex. Rules (array style):

| Field | Rules / notes |
| --- | --- |
| `name` | required, string, max:255 |
| `slug` | nullable, regex `^[a-z0-9]+(?:-[a-z0-9]+)*$`, `Rule::unique` ignoring own id |
| `phone` | **required**, regex requiring ≥6 digits, 6–30 chars |
| `currency` | required, `Rule::in(array_keys(config('currencies')))` |
| `default_locale` | required, `in:ar,en` |
| `logo_key` | conditionally required — `required` unless the restaurant already has a `logo` media item, else `nullable` (`RestaurantRequest.php:32-33,51`) |
| `cover_image_key`, `delete_logo`, `delete_cover_image`, `is_active`, `country_code`, `address`, `google_maps_url`, `description` | nullable/boolean/url helpers |

Messages are translation-driven (`__('menu_owner.restaurant.validation.*')`), unlike Category/Dish/Social which hard-code English strings.

### 5.5 Category management (`CategoryController`)

Standard CRUD plus a JSON `reorder`. Reads/writes the `Category` model scoped to `$request->user()->restaurant`.

- `index()` lists categories ordered by `name->{default_locale}` (a JSON-path order on the translatable name).
- `create()`/`store()` both bail to restaurant setup if no restaurant, then call `redirectIfCategoryLimitReached()` which consults `Restaurant::hasReachedCategoryLimit()` (entitlement check) and flashes `limit_categories` on overflow (`CategoryController.php:150-156`).
- `store()` (`CategoryController.php:52-75`) wraps the name/description strings into the base-locale translation via the private `translateContent()` helper, sets `restaurant_id`, computes `display_order = max(display_order)+1`, creates the row, then `syncImage()` → `MediaService::sync($category, image_key, delete_image, 'image', 'category-image')`.
- `edit()`/`update()`/`destroy()` each call `$this->authorize(... , $category)` → `CategoryPolicy`. `update()` re-uses `translateContent()` (passing the existing model so the other locale is preserved).
- `reorder()` (`CategoryController.php:113-129`) is a JSON endpoint: validates two integer ids `a`/`b`, fetches both scoped to the restaurant (`findOrFail`), swaps their `display_order`, flashes `reorder_success`, returns `{success:true}`.

**`CategoryRequest`** (array style, English messages): `authorize()` maps to `CategoryPolicy@update`/`@create` depending on whether a `Category` is route-bound. Rules: `name` required min:2 max:255; `description` nullable max:1000; `display_order` nullable-on-POST / required-on-PUT (method-aware via `isMethod('POST')`); `image_key` nullable string uuid; `delete_image` boolean.

### 5.6 Dish management (`DishController`)

Mirrors CategoryController but adds tag relations and a category dropdown.

- `create()`/`edit()` pass `categories` (ordered by translatable name), `tagValues` (existing tag ids), and `allTags` (`Tag::orderBy('category')`) to `dashboard/dishes/form.blade.php`.
- `store()` (`DishController.php:55-82`): setup/limit guard via `redirectIfDishLimitReached()` (`Restaurant::hasReachedDishLimit()` → `limit_dishes` flash), translate `name`/`ingredients` into base locale, **split `tags` out of the data array**, set `restaurant_id` + next `display_order`, `Dish::create()`, then `$dish->tags()->sync($tagIds)` (many-to-many) and `syncImage()` → `MediaService::sync($dish, dish_image_key, delete_dish_image, 'images', 'dish-image')`.
- `update()` authorizes (`DishPolicy`), re-translates against the existing dish, updates, re-syncs tags and image.
- `reorder()` behaves like the category version, scoped via `$restaurant->dishes()->findOrFail(...)`.

**`DishRequest`** (array style): `authorize()` → `DishPolicy@update`/`@create`. Notable rules: `category_id` required `exists:categories,id`; `name` required min:2 max:255; `price` nullable numeric 0–999999.99; `ingredients` nullable max:1000; `display_order` method-aware (nullable POST / required PUT); **`dish_image_key` required on create, nullable on update** (a photo is mandatory for new dishes — `DishRequest.php:30`, message "A dish photo is required."); `tags` nullable array max:30, `tags.*` integer `exists:tags,id`.

> **Variants note:** A `DishVariant` model exists (`app/Models/DishVariant.php`, translatable `name`, `price`, `is_available`, `display_order`) and a `dish_variants` table/migration is present, but these MenuOwner controllers and `DishRequest` do **not** read or write dish variants — there are no variant validation rules and no variant routes in `routes/web.php`. (The "variant" tokens in `dishes/form.blade.php` are the `<x-btn variant="…">` UI attribute, unrelated to dish variants.) Variant management is not part of this dashboard flow.

### 5.7 Authorization model (policies)

`CategoryPolicy`, `DishPolicy`, and `RestaurantSocialLinkPolicy` all follow the same ownership pattern: `create`/`viewAny` require admin or a non-null restaurant; `view`/`update`/`delete`/`restore`/`forceDelete` call `belongsToUserRestaurant()` which returns true for admins or when the record's `restaurant_id` equals the current user's restaurant id (`app/Policies/CategoryPolicy.php:45-54`, `DishPolicy.php:45-54`). This is what isolates one owner's menu from another's. Controllers enforce it via `$this->authorize('update'|'delete', $model)` on the mutating actions; the Form Requests independently re-check the same gates in `authorize()`.

### 5.8 Social links (`SocialLinkController`)

CRUD over `RestaurantSocialLink`. `index()` lists links ordered by `created_at`. `create`/`store` apply the setup guard and `Restaurant::hasReachedSocialLinkLimit()` (entitlement) → `limit_social` flash. `store()` injects `restaurant_id` and creates; `edit`/`update`/`destroy` authorize via `RestaurantSocialLinkPolicy`. **`SocialLinkRequest`**: `platform` required `in:instagram,x,facebook,tiktok,whatsapp`; `url` required, `url`, max:255, plus `regex:/^https?:\/\//i` (http/https only). Array-style rules, English messages.

### 5.9 AI menu scan (`MenuScanController`)

An asynchronous OCR import pipeline. Hard daily cap: `DAILY_SCAN_LIMIT = 3` (`MenuScanController.php:18`). Four actions:

1. **`index()`** — renders `dashboard/menu-scan/index.blade.php` with `scansToday` (count of today's `MenuScan` rows for the restaurant) and the limit.
2. **`scan()`** (`MenuScanController.php:40-84`) — validates `image` (required, image, max 10 MB, mimes jpeg/png/webp). Inside a `DB::transaction` it `lockForUpdate()`-counts today's scans; if `>= 3` returns HTTP **429** `{message:'daily_limit'}`. Otherwise creates a `MenuScan` (status `Pending`), attaches the upload to the Spatie `scan` collection (`local` disk, single file), and `ProcessMenuScan::dispatch($scan->id)`. Responds **202** with `scan_id` + status. The client then polls.
3. **`status()`** (`MenuScanController.php:89-104`) — owner-scoped `findOrFail` via `$restaurant->menuScans()`, returns `{status, result, error}`. The frontend polls this until `completed`/`failed`.
4. **`import()`** (`MenuScanController.php:109-218`) — confirms and persists the (possibly user-edited) extraction. Guards: scan must be `isCompleted()`, not already imported (`result['imported']`), and contain categories. It then **re-validates the edited payload** inline against the restaurant's live limits — `categories` max `category_limit`, each `categories.*.dishes` max `dish_limit`, names required ≤255, prices nullable 0–99999, ingredients ≤1000. It computes remaining slots and rejects (422) if the import would exceed dish or category limits. Persistence runs in a transaction that `lockForUpdate()`-re-checks the `imported` flag (throwing `already_imported` on a race), marks the scan imported, creates categories with `display_order` by array order, and **bulk `Dish::insert()`s** all dishes (names/ingredients JSON-encoded into the base locale, `is_available => true`). Returns a per-count success message.

The worker is **`App\Jobs\ProcessMenuScan`** (`app/Jobs/ProcessMenuScan.php`): `ShouldQueue`, `tries=3`, `timeout=90`, backoff `[60,120]` (to clear Gemini 429 windows). It marks the scan `processing`, reads the stored image, calls `GeminiMenuExtractorService::extractFromPath()`, then `markCompleted($result)` and **deletes the source image** (`clearMediaCollection('scan')`). Permanent failure → `markFailed()` + image cleanup. So `MenuScanController` is purely the HTTP entry/exit; the OCR itself is the job + the Gemini service.

### 5.10 QR code (`QrCodeController`)

Generates the public-menu QR with persisted styling. `DEFAULTS` = black FG, white BG, error-correction `M`.

- **`index()`** — builds `menuUrl` only when the restaurant is active and has a slug (`route('public.menu', slug)`), merges saved `qr_settings` over defaults, and pulls scan stats from `Restaurant::getQrScanCount()` for `total`/`month`/`week`/`today` (these count statistics rows where `via_qr = true`, `app/Models/Restaurant.php:236-246`). Renders `dashboard/qr-code/index.blade.php`.
- **`generate()`** (`QrCodeController.php:49-78`) — streams a live **SVG** via `simplesoftwareio/qrcode`. Query params `fg`/`bg` are run through `normalizeHex()` (validates `#RRGGBB`, falls back to saved/default), `ec` is whitelisted to `L/M/Q/H`. The encoded URL is the public menu with `?via=qr` so scans are attributable. Returns `image/svg+xml`. There is no Form Request here — colors/EC are sanitized inline.
- **`saveSettings()`** — JSON endpoint; inline-validates `fg_color`/`bg_color` (`regex /^#[0-9a-fA-F]{6}$/`) and `error_correction` (`in:L,M,Q,H`), persists `qr_settings` on the restaurant, flashes success.

### 5.11 Statistics (`StatisticController` + `StatisticsQueryService`)

`StatisticController` is intentionally thin (`StatisticController.php`): it reads the optional `?period` query param and delegates to `StatisticsQueryService::build($restaurant, $period)` (or `emptyPayload()` when there's no restaurant), spreading the result array straight into `dashboard/statistics/index.blade.php`.

`StatisticsQueryService::build()` (`StatisticsQueryService.php:14-129`) drives everything off `$restaurant->statistics()` (a `MenuView`/statistics relation). It whitelists `period` to `7/30/90/all` (default `30`), derives a previous comparable window, and computes: views-per-day series; period views / unique (`distinct session_id`) / WhatsApp orders with previous-period **percent deltas**; lifetime totals (`getTotalViews`, `getUniqueVisitors`, `getAverageTimeSpent`, `getTotalTimeSpent`, `getWhatsAppOrderCount`); page-view rollups for today/week/month; device/browser/OS breakdowns (`groupBy`, top 5); bounce rate (sessions with `time_spent < 10s`), average page-views/session, QR-attributed visits, and the 50 most recent rows. `emptyPayload()` returns the same keys zeroed so the Blade view never breaks pre-setup.

### 5.12 Settings (`SettingsController`)

Controls how the **public** menu renders. `DEFAULTS` cover nine toggles/options (`show_cover_image`, `show_prices`, `show_phone_number`, `show_address`, `show_social_links`, `show_ingredients`, `enable_share`, plus `share_button_position` and `menu_direction`). `index()` merges saved `template_settings` over defaults and renders `dashboard/settings/index.blade.php`. `update()` writes `$request->validated()` straight into `template_settings`.

**`SettingsRequest`**: `authorize()` allows owners/admins. Because HTML checkboxes omit unchecked values, `prepareForValidation()` coerces all seven booleans via `$this->boolean(...)` before validation (`SettingsRequest.php:31-42`). Rules: each toggle `boolean`; `share_button_position` `in:bottom_right,bottom_left,top_right,top_left`; `menu_direction` `in:ltr,rtl`.

### 5.13 Search (`SearchController`)

A JSON typeahead for the owner's own data (`SearchController.php`). Trims `q`; on empty returns `{dishes:[],categories:[]}`. Otherwise `LIKE`-matches both `name->ar` and `name->en` (translatable JSON columns), scoped to the restaurant, ordered by `name->{default_locale}`, capped at 6 dishes / 4 categories. Returns minimal columns (`id`, `name`, `is_available` / `name`).

### 5.14 Temp upload (`TempUploadController`)

The deferred-upload entry point that makes the image flow work without immediately attaching files to records. A single `store()` action delegates to `MediaService::storeTempUpload(file, context, userId)` and returns its JSON (`{key, original_size, optimized_size, saved_percent}`). The returned `key` (a UUID) is what category/dish/restaurant forms post back as `image_key`/`dish_image_key`/`logo_key`/`cover_image_key`, and `MediaService::sync()` promotes it on save. Route is `throttle:uploads`-limited.

**`TempUploadRequest`**: `authorize()` only requires an authenticated user. Rules: `file` required, `file`, `image`, `mimes:jpeg,jpg,png,webp`, `max:10240` (10 MB); `context` nullable `in:logo,cover_image,dish,category,generic` (selects the optimization preset in `MediaService::preset()`). Friendly English messages for each failure.

### 5.15 Profile (`ProfileController`)

## 6. Entitlements, Subscriptions & Statistics Services

This section covers two related back-end subsystems: the **entitlements/monetization layer** (templates as plans, features as sellable units, subscriptions and per-restaurant grants resolved into effective limits and capabilities) and the **statistics pipeline** (raw menu sessions rolled up nightly into a daily table, then aggregated for the owner dashboard and admin widgets).

### 6.1 Entitlements service

`App\Services\MenuOwner\Entitlements` (`app/Services/MenuOwner/Entitlements.php`) is the single source of truth for "what is this restaurant allowed to do, and how much." It wraps one `Restaurant` and exposes three public reads plus a cache-flush helper:

| Method | Returns | Purpose |
| --- | --- | --- |
| `Entitlements::for($restaurant)` | `self` | Static factory (used by `Restaurant::entitlements()`). |
| `can(string $slug)` | `bool` | True if a boolean capability is granted (defaults to `false`). |
| `limit(string $slug)` | `int` | Effective numeric limit; falls back to `config('entitlements.defaults.{slug}', 0)` when not set (`Entitlements.php:27-34`). |
| `all()` | `array<string,bool\|int>` | Full resolved map, memoised on the instance and in the cache. |
| `Entitlements::flush($restaurantId)` (static) | `void` | Forgets the cached map for one restaurant. |

#### Resolution algorithm (`resolve()`, `Entitlements.php:69-107`)

The effective entitlement map is built by layering three sources, in order, and merging each feature in:

1. **Template bundle** — the restaurant's assigned `template->features` (the `template_feature` pivot, each row carrying a `value`). The bundle is only applied if the template `isFree()` **or** the restaurant has an `activeTemplateSubscription()` (`Entitlements.php:75-79`). A paid template with no active subscription contributes nothing.
2. **Feature add-on subscriptions** — active subscriptions whose `subscribable_type` is `Feature::class`. Each grants its feature with value `'1'` (`Entitlements.php:81-92`).
3. **Per-restaurant feature grants** (`restaurant_features` rows via `featureGrants()`) that are currently valid: `starts_at <= now()` and (`ends_at` is null or in the future). Each applies its stored `value` (`Entitlements.php:94-104`).

Merging is done by `merge()` (`Entitlements.php:113-122`) according to the feature's `kind`:
- **`limit`** features merge with **MAX** — the highest value across all sources wins.
- **boolean** features merge with **OR** — granted if any source grants it.

This means add-ons and grants can only ever *raise* a limit or *enable* a capability, never lower one set by the template.

#### Caching

`all()` memoises onto `$this->resolved` and also caches the resolved array under key `entitlements:{restaurantId}` for `config('entitlements.cache_ttl', 300)` seconds (`Entitlements.php:43-54, 61-64`). The cache is invalidated through `Entitlements::flush()`, which is wired into model `booted()` hooks so writes are self-healing:
- `Restaurant` flushes on save **only when `template_id` changed** (`app/Models/Restaurant.php:66-70`).
- `Subscription` flushes on every `saved`/`deleted` (`app/Models/Subscription.php:46-52`).
- `RestaurantFeature` flushes on every `saved`/`deleted` (`app/Models/RestaurantFeature.php:32-38`).

`EntitlementsTest` calls `Entitlements::flush()` manually after directly creating grant rows (e.g. `EntitlementsTest.php:115`) because those rows are created through the relationship, confirming the flush-on-write contract.

#### Configuration (`config/entitlements.php`)

| Key | Value | Meaning |
| --- | --- | --- |
| `defaults.dish_limit` | `40` | Floor dish limit when no template/grant sets it. |
| `defaults.category_limit` | `10` | Floor category limit. |
| `defaults.social_link_limit` | `4` | Floor social-link limit. |
| `grace_days` | `7` | Days a `past_due` subscription keeps entitling after `current_period_end`. |
| `cache_ttl` | `300` | Seconds the resolved map is cached. |

The comment in the config notes booleans default to `false` (no `defaults` entry needed) while limits need an explicit floor (`config/entitlements.php:16-20`).

#### How the `Restaurant` model exposes entitlements

`Restaurant` (`app/Models/Restaurant.php`) is the consumer-facing surface. It defines:
- `entitlements(): Entitlements` (`:138-141`) — returns the service for this restaurant.
- Magic accessors that read limits: `getDishLimitAttribute()`, `getCategoryLimitAttribute()`, `getSocialLinkLimitAttribute()` map `$restaurant->dish_limit` etc. to `entitlements()->limit(...)` (`:171-184`).
- Limit-check helpers: `hasReachedDishLimit()`, `getRemainingDishSlots()`, and the category/social-link equivalents compare current counts against the entitlement limit (`:186-214`).
- `activeTemplateSubscription()` (`:143-155`) — the latest active subscription whose `subscribable` matches the assigned template, using the `active()` scope.
- `menuIsPubliclyAvailable()` (`:162-169`) — the gate that takes a menu offline: returns `true` if there is no template or the template is free, otherwise requires an active (or in-grace) template subscription. Expiry takes the menu offline without deleting data.

#### How controllers query entitlements

Limit enforcement lives in the menu-owner controllers, not in the Form Requests (which only validate field shape). The pattern is a guard at `create`/`store`:

| Controller | Guard | On limit reached |
| --- | --- | --- |
| `DishController` (`app/Http/Controllers/MenuOwner/DishController.php:168-174`) | `redirectIfDishLimitReached()` → `hasReachedDishLimit()` | redirect to index with `limit_dishes` error |
| `CategoryController` (`app/Http/Controllers/MenuOwner/CategoryController.php:42-44`) | `redirectIfCategoryLimitReached()` → `hasReachedCategoryLimit()` | redirect with `limit_categories` error |
| `SocialLinkController` (`app/Http/Controllers/MenuOwner/SocialLinkController.php:54-57`) | `hasReachedSocialLinkLimit()` | redirect with `limit_social` error |

`menuIsPubliclyAvailable()` is the capability-style read used at the public boundary (the menu goes offline when a paid plan lapses). `EntitlementsTest` verifies the full matrix: config fallbacks without a template, free-template bundles, paid-without-subscription (offline + falls back to defaults), paid-with-subscription (online + bundle limits), `past_due` within grace (still online), expired (offline), grant overlay with MAX merge, and expired-grant exclusion (`tests/Feature/EntitlementsTest.php`).

### 6.2 Subscription & pricing model

The monetization schema is created in one migration, `database/migrations/2026_06_12_134516_create_entitlement_tables.php`, whose header states the design intent: *templates are the plans, features are the sellable units (booleans or limits), bundled via `template_feature` or sold as add-ons; subscriptions/payments support manual billing first.*

| Table / Model | Role | Key columns |
| --- | --- | --- |
| `templates` / `Template` | A plan/tier. | `tier` (`free`/`paid`), `capabilities` (json), `default_settings` (json), `is_active`, `sort_order` |
| `features` / `Feature` | A sellable unit. | `slug`, `kind` (`boolean` or `limit`, default `boolean`), `is_addon`, `is_active` |
| `template_feature` (pivot) | Bundles features into a template with a per-template value. | composite PK `(template_id, feature_id)`, `value` (string, default `'1'`) |
| `template_prices` / `TemplatePrice` | Per-period price for a template. | `period`, `price` (decimal 10,2), `currency`, `is_active` |
| `feature_prices` / `FeaturePrice` | Per-period price for an add-on feature. | `period`, `price`, `currency`, `is_active` |
| `subscriptions` / `Subscription` | A restaurant's active plan or add-on (polymorphic). | `morphs('subscribable')`, `period`, `price`, `currency`, `status`, `starts_at`, `current_period_end`, `trial_ends_at`, `canceled_at`, `provider`, `provider_ref`, `meta` |
| `payments` / `Payment` | Polymorphic payment record. | `morphs('payable')`, `amount`, `status`, `provider` (default `manual`), `paid_at`, `meta` |
| `restaurant_features` / `RestaurantFeature` | Per-restaurant feature grant. | `value`, `source` (default `purchase`, e.g. `grant`), `starts_at`, `ends_at`, `payment_id` |

#### Model relationships

- `Template` (`app/Models/Template.php`): `features()` (belongsToMany on `template_feature` with `withPivot('value')`), `prices()` (hasMany `TemplatePrice`), `subscriptions()` (morphMany), plus helpers `isFree()` / `isPaid()` (tier-based, default free), `capability($key, $default = true)` reading the `capabilities` json, and `resolveSettings()` merging stored owner overrides over `default_settings`.
- `Feature` (`app/Models/Feature.php`): `prices()` (hasMany `FeaturePrice`), `templates()` (belongsToMany), `isLimit()` (`kind === 'limit'`). Both `Template` and `Feature` are translatable (`name`, `description`).
- `Subscription` (`app/Models/Subscription.php`): `restaurant()`, `subscribable()` (morphTo, points at a `Template` or `Feature`), `payments()` (morphMany on `payable`).

#### What counts as "active" / entitling

The entitling window is defined once and reused. `Subscription::scopeActive()` (`Subscription.php:73-86`) and the per-row `isEntitling()` (`:88-97`) both encode the same rule:
- `trialing` or `active` **and** `current_period_end` is in the future, **or**
- `past_due` **and** `current_period_end > now()->subDays(grace_days)` (the 7-day grace window from config).

The `subscriptions` table indexes `status` and `(restaurant_id, status)` to keep this scope fast. `Entitlements::resolve()` and `Restaurant::activeTemplateSubscription()` both call `subscriptions()->active()`, so the menu-offline gate and the entitlement map stay consistent.

### 6.3 Statistics pipeline

The analytics flow has three stages: raw per-visit rows → nightly daily rollup → aggregation for display.

#### Stage 1 — Raw menu sessions

Raw visit rows live in the `menu_sessions` table, mapped by `App\Models\RestaurantStatistic` (`app/Models/RestaurantStatistic.php`; note `protected $table = 'menu_sessions'`). The migration `2026_06_12_134520_rename_statistics_to_menu_sessions_and_create_daily_rollup.php` renamed the old `restaurant_statistics` table to `menu_sessions` and created the rollup table. Raw columns: `session_id`, `device_type`, `browser`, `os`, `viewed_at`, `time_spent`, `page_views`, `via_qr` (bool), `whatsapp_orders`.

These rows are written by the **public menu** controller, `App\Http\Controllers\Portal\MenuController`:
- `trackVisitor()` (`:92-115`) creates one row per session per day (device/browser/OS detected from the user agent, `via_qr` set when `?via=qr`), or increments `page_views` on the existing row for that session+day.
- `trackExit()` (`:74-90`) raises `time_spent` to the max reported value (fired via `navigator.sendBeacon` on page exit; CSRF-exempt — see below).
- `trackWhatsAppOrder()` (`:57-72`) increments `whatsapp_orders` on today's session row.

The track-exit and track-whatsapp endpoints are CSRF-exempt because they fire on page unload after the session token may have expired (`bootstrap/app.php:29-32`, excluding `*/track-exit` and `*/track-whatsapp-order`).

#### Stage 2 — Nightly rollup (`stats:rollup`)

`App\Console\Commands\RollupMenuSessions` (`app/Console/Commands/RollupMenuSessions.php`, signature `stats:rollup`) buckets raw sessions into `RestaurantDailyStat` (`restaurant_stats_daily`). Options: `--days=2` (trailing days to recompute per restaurant) and `--prune-months=6` (delete raw `menu_sessions` older than N months).

Key logic:
- It chunks restaurants by 100 and, for each, computes buckets **in the restaurant's own timezone** (`$restaurant->timezone ?: 'UTC'`) so "Tuesday" means Tuesday where the restaurant lives, not UTC (`RollupMenuSessions.php:43-50`). Each local day is converted to a UTC `[startOfDay, endOfDay]` window to select rows.
- For each day with rows it `updateOrCreate`s a daily row keyed on `(restaurant_id, date)` with: `visits` (row count), `unique_sessions` (distinct `session_id`), `qr_visits` (`via_qr = true`), `whatsapp_orders` (sum), and `avg_time_spent` (rounded average over rows with `time_spent > 0`, or null) (`:62-71`).
- After rollup, if `--prune-months > 0`, it deletes raw `menu_sessions` rows older than that cutoff (`:29-32`).

`RestaurantDailyStat` (`app/Models/RestaurantDailyStat.php`) maps `restaurant_stats_daily` (unique `(restaurant_id, date)`) with integer-cast aggregate columns and a `restaurant()` belongsTo. `Restaurant::dailyStats()` is the hasMany. The rollup table is the intended long-term store; raw rows are pruned.

**Scheduling:** registered in `routes/console.php:12` as `Schedule::command('stats:rollup')->dailyAt('03:10')` (Laravel 12 registers `routes/console.php` for scheduling via `bootstrap/app.php`; there is no `Console\Kernel`).

#### Stage 3 — Aggregation for display

`App\Services\MenuOwner\StatisticsQueryService` (`app/Services/MenuOwner/StatisticsQueryService.php`) builds the owner dashboard payload. Note it currently queries the **raw `menu_sessions`** via `$restaurant->statistics()`, not the daily rollup (the rollup table exists and is populated, but the dashboard still reads raw rows).

`build($restaurant, $periodInput)` (`:14-129`):
- Validates `$periodInput` to one of `7`/`30`/`90`/`all`, defaulting to `30` (`:16`). For `all`, the period start is the restaurant's `created_at` (`:18-25`).
- Computes a matching **previous period** window and percentage deltas via a `$pct` closure (null when the previous value is 0) for views, unique visitors, and WhatsApp orders (`:42-52`).
- Produces: `visitsPerDay` (grouped `DATE(viewed_at)`), period and lifetime totals (`totalViews`, `uniqueVisitors`, `averageTimeSpent`, `totalTimeSpent`, `totalPageViews`), `viewsToday`/`viewsThisWeek`/`viewsThisMonth`, `bounceRate` (sessions with `time_spent < 10s` over unique visitors), `avgPageViewsPerSession`, `qrVisits`, `whatsappOrders`, and breakdowns by `device_type`, top-5 `browser`, top-5 `os`, plus the 50 most recent rows and the public `menuUrl` (`:54-128`). Several lifetime totals delegate to `Restaurant` helpers (`getTotalViews`, `getUniqueVisitors`, `getAverageTimeSpent`, `getTotalTimeSpent`, `getWhatsAppOrderCount`).
- `emptyPayload()` (`:136-165`) returns a fully zeroed/empty structure for owners who have no restaurant yet.

**Consumer:** `App\Http\Controllers\MenuOwner\StatisticController::index()` (`app/Http/Controllers/MenuOwner/StatisticController.php`) injects the service, reads the `period` query param, and renders `dashboard.statistics.index` with either `build(...)` or `emptyPayload()`. `StatisticsQueryServiceTest` asserts the aggregation math (totals, unique visitors, QR visits, WhatsApp sums, device breakdown) and the zeroed empty payload (`tests/Feature/StatisticsQueryServiceTest.php`).

**Admin widgets** read raw sessions directly rather than through the service:
- `PopularRestaurantsWidget` (`app/Filament/Admin/Widgets/PopularRestaurantsWidget.php`) ranks restaurants by `withSum('statistics', 'page_views')` and `withCount('statistics')`.
- `RecentActivityWidget` (`app/Filament/Admin/Widgets/RecentActivityWidget.php`) lists the 15 latest `RestaurantStatistic` rows with device/browser/OS columns.

Other admin widgets (`VisitorStatsWidget`, `TodayOverviewWidget`) and the `RestaurantStatisticResource` also consume the raw `menu_sessions` model.

## 7. Public Portal (Customer-Facing Menu)

The public portal is the unauthenticated, customer-facing surface of qayema. It comprises three concerns: (1) the **marketing/landing site** (home, contact, legal pages), and (2) the **live restaurant menu** reached by scanning a QR code, plus (3) the **analytics tracking** that records every visit. All of these are registered in `routes/web.php` and are guarded only by rate limiting and a session-resolved locale — there is no authentication.

### 7.1 Routing overview

Public routes live at the bottom of `routes/web.php` and are deliberately ordered so the catch-all `/{slug}` menu route is the **last** route registered, after all named owner/auth/legal routes. The slug routes constrain the parameter to `[a-z0-9-]+` so numeric or uppercase paths never match.

| Method | URI | Name | Controller / action | Middleware |
|---|---|---|---|---|
| GET | `/` | (none) | inline closure → `view('portal.welcome')` | `owner.locale` |
| GET | `/locale/{locale}` | `locale.switch` | inline closure (sets `owner_locale` session, redirects back) | `owner.locale` |
| GET | `/privacy-policy` | `privacy` | closure → `view('legal.privacy')` | `owner.locale` |

### 7.2 How a visitor reaches a restaurant menu (slug resolution)

`Portal\MenuController::show()` (`app/Http/Controllers/Portal/MenuController.php:19`) is the entry point. Resolution steps:

1. **Slug guard** — `guardSlug()` (`MenuController.php:117`) aborts 404 if the slug is in `RESERVED_PATHS` (`categories`, `dishes`, `dashboard`, `restaurant`, `settings`, `statistics`, `qr-code`, `social-links`, `profile`, `admin`, `login`, `register`, `logout`, `password`, `email`, `sanctum` — `MenuController.php:15`) or is fully numeric. This prevents the catch-all route from shadowing real app paths and is verified by `test_numeric_slug_is_rejected` in `tests/Feature/Portal/MenuControllerTest.php`.
2. **Restaurant lookup** — `Restaurant::where('slug', $slug)->where('is_active', true)->firstOrFail()` (`MenuController.php:23-32`). An inactive or unknown restaurant produces a 404 (`test_inactive_restaurant_returns_404`, `test_unknown_slug_returns_404`). The query eager-loads `template`, `socialLinks`, ordered `categories`, each category's **available** dishes (`is_available = true`, ordered by `display_order`), and uncategorized available dishes (`whereNull('category_id')`) — avoiding N+1 when rendering.
3. **Paywall / "temporarily unavailable" gate** — if `Restaurant::menuIsPubliclyAvailable()` (`app/Models/Restaurant.php:162`) returns false, the controller returns `response()->view('portal.menu.unavailable', ...)` with **HTTP 200** (`MenuController.php:37-39`). `menuIsPubliclyAvailable()` returns true for free templates or no template, and otherwise requires an active template subscription. The 200 status is intentional so a **printed QR code never breaks**; the data is untouched and a renewal restores the menu. This is covered by `tests/Feature/Portal/MenuOfflineTest.php` (`test_paid_template_without_subscription_serves_unavailable_page_with_200` asserts 200, view `portal.menu.unavailable`, and `menu_sessions` count 0; the active-subscription and free-template cases serve `portal.menu.designs.default`).
4. **Visitor tracking** — `trackVisitor()` is called only when the menu is actually served (`MenuController.php:41`), so the unavailable page records no statistic.
5. **Settings merge & render** — template settings are resolved by merging `config('menu_settings.defaults')` with `Restaurant::resolvedTemplateSettings()` (which delegates to `Template::resolveSettings()`, `Restaurant.php:270`). The final view is `view('portal.menu.designs.default', [...])` passing `restaurant`, `categories`, `uncategorizedDishes`, `settings`, and a `template` slug (`MenuController.php:48-54`).

> Note on the rendered menu view: the controller targets `portal.menu.designs.default` and `portal.menu.unavailable`, but **those Blade files are not present in `resources/views/portal/menu/`** in the current tree (only the menu **components** `resources/views/components/menu/cart.blade.php` and `resources/views/components/menu/dish-card.blade.php` exist). Tests assert the view *names* via `assertViewIs(...)` (which does not require the file to be rendered to a known view in every assertion path), so the design templates appear to be pending/un-committed. The cart and dish-card components below are what a menu design view is expected to compose.

### 7.3 Menu UI components and the WhatsApp ordering flow

- **`components/menu/dish-card.blade.php`** renders one dish. It reads display toggles from `$settings` (`show_prices`, `show_dish_image`, `show_ingredients`), shows the image (lazy-loaded) only if present, formats price via `Number::currency()`, and on click/Enter/Space calls `Alpine.store('cart').add({...})` with the dish id, name, price, and image URL.
- **`components/menu/cart.blade.php`** defines an Alpine.js `cart` store (sidebar drawer, item list, quantity controls, currency formatting via `Intl.NumberFormat`). Its `checkout()` method (`cart.blade.php:271`) is the WhatsApp-order path: it first fires `fetch('/${slug}/track-whatsapp-order', { method:'POST', headers:{ 'X-CSRF-TOKEN', 'Accept': application/json } })` (tracking failure is swallowed as non-fatal), then builds a formatted order message and opens `https://wa.me/${phone}?text=${encoded}` in a new tab. The phone is stripped to digits with `replace(/\D/g,'')`.

### 7.4 Statistics tracking (views, scans, WhatsApp clicks, time-on-page)

All raw per-visit rows are written to the **`menu_sessions`** table via the `RestaurantStatistic` model (`app/Models/RestaurantStatistic.php:17` sets `protected $table = 'menu_sessions'`; the model docblock notes this is a v2 rename and that dashboards will eventually read a `restaurant_stats_daily` rollup). Tracking is session-scoped per day:

| Event | Trigger | Method | Column written |
|---|---|---|---|
| First view of the day | GET `/{slug}` (menu served) | `trackVisitor()` `MenuController.php:92` → creates a row | `page_views = 1`, `device_type`, `browser`, `os`, `viewed_at`, `via_qr` |
| Repeat view same day/session | GET `/{slug}` again | `trackVisitor()` finds existing row → `increment('page_views')` | `page_views++` |
| QR-sourced scan | GET `/{slug}?via=qr` | `trackVisitor()` sets `'via_qr' => $request->query('via') === 'qr'` `MenuController.php:113` | `via_qr = true` |
| WhatsApp order click | POST `/{slug}/track-whatsapp-order` | `trackWhatsAppOrder()` `MenuController.php:57` → `increment('whatsapp_orders')` on today's row | `whatsapp_orders++` |
| Page exit / time spent | POST `/{slug}/track-exit` | `trackExit()` `MenuController.php:74` → `update(['time_spent' => max(existing, posted)])` | `time_spent` (max kept) |

The "today's row" for a session is found by `findTodayStatistic()` (`MenuController.php:124`): `restaurant_id` + `session_id` + `whereDate('viewed_at', today())`, newest first. The `track-exit` and `track-whatsapp-order` endpoints are **CSRF-exempt** (`bootstrap/app.php:29-32`, patterns `*/track-exit` and `*/track-whatsapp-order`) because `track-exit` is fired via `navigator.sendBeacon` on page unload, which can run after the CSRF token/session has expired; exempting them avoids spurious 419s. `test_visiting_a_menu_records_a_statistic` asserts a `menu_sessions` row is created and exactly one `RestaurantStatistic` exists after one visit.

### 7.5 Device detection

`Portal\DeviceDetectionService` (`app/Services/Portal/DeviceDetectionService.php`) is constructor-injected into `MenuController` and called during `trackVisitor()` to classify the User-Agent string into three fields stored on the statistic row:

- `getDeviceType()` — regex-based: `tablet|ipad|playbook|silk` → `tablet`; `mobile|android|iphone|ipod|blackberry|iemobile|opera mini` → `mobile`; else `desktop`; null UA → `unknown`.
- `getBrowser()` — substring checks with correct precedence (`edg` → Edge before `chrome`; `opr`/`opera` → Opera; then Chrome, Firefox, Safari; else `Unknown`).
- `getOS()` — `android`→Android, `iphone`/`ipad`→iOS, then Windows, macOS, Linux; else `Unknown`.

These feed the device/browser/OS breakdowns shown in the owner's statistics dashboard.

### 7.6 Offline / PWA behavior

There is **no PWA / service worker / offline-caching layer** in the portal — no `manifest.json`, no service worker registration, and no offline asset bundle exist in `public/` or the portal views. The only "offline-like" concept is the **paywall-driven "temporarily unavailable" page** described in §7.2: when a paid template lacks an active subscription, the menu is served as `portal.menu.unavailable` with HTTP 200 so QR codes keep resolving while the menu is dormant. The test file that covers this is named `MenuOfflineTest.php`, which is what "offline" refers to in this codebase (menu offline = temporarily not published), **not** browser offline support.

### 7.7 Contact form flow (throttling + captcha + honeypot)

The contact form is the most defended public endpoint. The page is served by `ContactController::show()` → `view('portal.contact')` and submitted to `ContactController::store()` (`app/Http/Controllers/ContactController.php`).

**Layered abuse defenses (in order):**

1. **Route-level rate limit** — `throttle:contact` (10/min, no auto-ban).
2. **Honeypot** — the form includes a visually-hidden `hp_field` input (`resources/views/portal/contact.blade.php:82-85`, off-screen, `autocomplete=off`, password-manager opt-out attributes). If `filled($request->input('hp_field'))`, the controller **fakes success** and silently drops the submission (`ContactController.php:22-26`) — bots get no signal.
3. **Validation** via `ContactRequest` (`app/Http/Requests/ContactRequest.php`): `name` required ≤100, `email` required/valid ≤255, `message` required 10–2000 chars. `authorize()` returns true (public, open to guests). A `g-recaptcha-response` field is validated by the `ValidCaptcha` rule.
4. **Captcha** — `App\Rules\ValidCaptcha` calls `App\Services\ThirdParty\Captcha::verify($token, Request::ip())`. `Captcha::verify()` short-circuits to `true` when `config('services.recaptcha.enabled')` is false; otherwise it POSTs the token to Google's siteverify endpoint (3s connect / 5s timeout, failures treated as invalid). The Blade form only injects the hidden token input and loads the reCAPTCHA v3 script when `config('services.recaptcha.enabled')` is true, refreshing the `action: 'contact'` token every 110s.
5. **Durable per-IP daily limit** — enforced in the service, see below.

**Submission service** — `App\Services\Portal\ContactService::submit()` (`app/Services/Portal/ContactService.php:51`):
- Throttle check: `isThrottled($ip)` counts `contact_messages` rows for that IP in the trailing 24h; the cap is `MAX_PER_DAY = 3` (`ContactService.php:15`). If reached, it throws `TooManyContactMessages($this->availableInHours($ip))`. This quota is deliberately enforced **against the `contact_messages` table, not the cache**, so it survives `cache:clear`, deploys, and queue restarts (documented at `ContactService.php:43-49`).
- `availableInHours()` (`ContactService.php:27`) computes hours until the oldest of the recent messages ages out (oldest `created_at` + 1 day), clamped to a minimum of 1.
- On success, `createAndNotify()` persists a `ContactMessage` (`app/Models/ContactMessage.php`, fillable: `name`, `email`, `message`, `ip_address`) and dispatches the mailable. The mail dispatch is wrapped in try/catch and only **logged** on failure — because the message is already saved, a queue hiccup must not fail the user's submission (`ContactService.php:84-94`).

**The throttle exception** — `App\Exceptions\TooManyContactMessages` (`app/Exceptions/TooManyContactMessages.php`) is a plain exception carrying a public readonly `int $retryAfterHours`. The controller catches it and renders a localized message `__('portal.contact.js.rate_limit', ['hours' => ...])`; for AJAX it returns a **429 JSON** error keyed `rate_limit`, and for no-JS it does `back()->withInput()->withErrors(['rate_limit' => ...])` (`ContactController.php:30-38`). Because locale is resolved server-side by the `owner.locale` middleware, the message is already translated for both paths.

**Notification mail** — `App\Mail\ContactMessageReceived` (`app/Mail/ContactMessageReceived.php`) implements `ShouldQueue` (so `Mail::to(...)->send(...)` only enqueues a job, no SMTP in-request). Its envelope sets `replyTo` to the submitter's email/name and a subject "New contact message from {name}"; the body is the Markdown view `emails.contact-received`. The recipient address comes from `config('services.contact.recipient')`.

**Dual-mode UX** — `portal/contact.blade.php` supports both no-JS and AJAX. Without JS, a server redirect with `session('success')` reveals the success panel and `$errors` populate inline. With JS, an inline script does client-side validation (mirroring the server rules), a live character counter (`/ 2000`), `fetch()` submission with `X-Requested-With: XMLHttpRequest`, and maps any returned `errors` (including `rate_limit` and `g-recaptcha-response`) into field errors / a banner. Validation/UI strings are injected server-side via `@json(__('portal.contact.js'))` so they are already in the active locale.

### 7.8 Legal pages and localization

## 8. AI Menu Scanning, Media & Mail

This section covers three intertwined infrastructure concerns: the AI menu-scan pipeline (photograph a paper menu → Gemini extracts structured categories/dishes → owner reviews and imports), the reusable image media pipeline (Intervention Image optimization + Spatie Media Library), and the transactional/queued email flows.

### 8.1 AI Menu Scanning — End-to-End Pipeline

The scan flow is deliberately asynchronous: the HTTP request returns immediately (`202 Accepted`) and the browser polls a status endpoint while a queued job calls Gemini in the background. The owner then reviews/edits the extracted data and confirms a separate "import" step that actually writes `Category` and `Dish` rows.

#### Pipeline stages

| Stage | Component | Outcome |
| --- | --- | --- |
| 1. Upload | `MenuScanController::scan()` (`app/Http/Controllers/MenuOwner/MenuScanController.php:40`) | Validates image, enforces daily limit, creates `MenuScan` (status `pending`), attaches image to `scan` media collection, dispatches job, returns `202` with `scan_id` |
| 2. Background extraction | `ProcessMenuScan` job (`app/Jobs/ProcessMenuScan.php`) | Marks `processing`, reads the image off the `local` disk, calls Gemini, marks `completed` (with result) or `failed` (with error). Deletes the image afterward |
| 3. Gemini call | `GeminiMenuExtractorService::extractFromPath()` (`app/Services/ThirdParty/GeminiMenuExtractorService.php:33`) | Base64-encodes image, POSTs to Gemini `generateContent`, parses + normalizes the structured JSON response |
| 4. Polling | `MenuScanController::status()` (`MenuScanController.php:89`) | Owner-scoped lookup returning `status`, `result`, `error` |
| 5. Import | `MenuScanController::import()` (`MenuScanController.php:109`) | Validates the (possibly edited) categories, enforces plan limits, bulk-inserts categories/dishes inside a transaction, marks result `imported` to be idempotent |

#### Routes

All routes live in the menu-owner group (`routes/web.php:96-100`), which is gated by `auth`, onboarding/setup-complete middleware, `owner.locale`, and `EnsureUserIsMenuOwner` + `throttle:mutations`. The two write endpoints add `throttle:uploads`.

| Method | URI | Name | Action |
| --- | --- | --- | --- |
| GET | `/menu-scan` | `menu-owner.menu-scan.index` | Renders `dashboard.menu-scan.index`, passing `scansToday` and `scanLimit` |
| POST | `/menu-scan` | `menu-owner.menu-scan.scan` | Accept upload, create scan, dispatch job |
| GET | `/menu-scan/{id}/status` | `menu-owner.menu-scan.status` | Poll for status/result/error |
| POST | `/menu-scan/{id}/import` | `menu-owner.menu-scan.import` | Confirm + write categories/dishes |

#### `MenuScan` model & status enum

`app/Models/MenuScan.php` is a Spatie `HasMedia` model with a single-file `scan` media collection on the **`local`** disk, restricted to `image/jpeg`, `image/png`, `image/webp` (`MenuScan.php:27-33`). It defaults `status` to `pending` (`$attributes`), casts `status` to `MenuScanStatus` and `result` to `array`, and `belongsTo` `Restaurant`. The table (`database/migrations/2026_05_08_134432_create_menu_scans_table.php`) holds `restaurant_id` (cascade delete), `status`, `result` (JSON), `error` (text), timestamps, and an index on `[restaurant_id, status]`. The original `image_path` column was dropped in `2026_06_12_134521_move_menu_scan_images_to_media.php` once uploads moved into the media library.

State transitions are encapsulated as methods rather than raw updates:

| Method | Effect |
| --- | --- |
| `markProcessing()` | status → `processing`, clears `error` |
| `markCompleted(array $result)` | status → `completed`, stores `result` |
| `markFailed(string $error)` | status → `failed`, stores `error` |
| `isPending()` / `isCompleted()` / `isFailed()` | enum comparisons |

`MenuScanStatus` (`app/Enums/MenuScanStatus.php`) is a string-backed enum: `Pending`, `Processing`, `Completed`, `Failed`.

#### Upload & daily-limit enforcement (`scan()`)

- Validation: `image` is `required|image|max:10240` (10 MB) `|mimes:jpeg,png,webp` (`MenuScanController.php:42-44`).
- A `private const DAILY_SCAN_LIMIT = 3` caps scans per restaurant per calendar day.
- The create + limit check run inside `DB::transaction` with `lockForUpdate()` on today's count, so concurrent uploads can't slip past the cap (`MenuScanController.php:52-72`). Hitting the cap returns `null` from the closure → `429` with `{"message":"daily_limit","limit":3}`.
- On success: `MenuScan::create(...)` then `addMediaFromRequest('image')->usingName('menu-scan')->toMediaCollection('scan')`, `ProcessMenuScan::dispatch($scan->id)`, and a `202` with `scan_id` + `status`.

#### `ProcessMenuScan` job

Queued job (`implements ShouldQueue`, uses `Queueable`) configured with `tries = 3`, `timeout = 90`, and `backoff() = [60, 120]` seconds — the comment notes this is sized to clear Gemini's 429 rate-limit window (`ProcessMenuScan.php:17-27`). It takes only `private readonly int $menuScanId` and re-fetches the record in `handle()`.

`handle(GeminiMenuExtractorService $extractor)` logic (`ProcessMenuScan.php:29`):
1. Logs start with attempt count; returns early if the record is missing or already `completed` (idempotency guard against retries/duplicate dispatch).
2. `markProcessing()`.
3. Resolves the image path via `getFirstMedia('scan')?->getPath()`. If missing/absent on disk → `markFailed('The uploaded image is no longer available...')`, clears media, returns.
4. Calls `extractFromPath()`. A `RuntimeException` is logged (with `tries_left`) and **re-thrown**, so Laravel's retry machinery applies the backoff and eventually routes to `failed()`.
5. On success, logs counts, `markCompleted($result)`, and deletes the image.

`failed(Throwable)` (`ProcessMenuScan.php:99`) runs after all retries are exhausted: logs the permanent failure, `markFailed()` with the exception message (or a generic fallback), and clears the `scan` media collection. The private `deleteImage()` (`clearMediaCollection('scan')`) is called on every terminal path (success, file-missing, permanent failure), so menu photos are never retained after processing.

#### `GeminiMenuExtractorService`

Constructed from config: `services.gemini.api_key` and `services.gemini.model` (default in code `gemini-1.5-flash`, default in config `gemini-2.0-flash`). It targets the REST endpoint:

```
https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent?key={apiKey}
```

The HTTP call uses `Http::timeout(60)->retry(2, 5000, throw: false)` (`GeminiMenuExtractorService.php:47`). `ConnectionException` is caught and converted to a friendly `RuntimeException`.

**Request payload** (`buildPayload()`, `GeminiMenuExtractorService.php:157`): one `contents` entry with two parts — `inline_data` (base64 image + detected MIME type) and a text instruction asking it to "Extract all categories and dishes... name, price as a number (null if not shown), and any description or ingredients as a single plain text string... Group dishes under their category." Critically it sets `generationConfig.response_mime_type = application/json` plus a strict `response_schema` (Gemini structured-output mode) describing the shape:

```
{ categories: [ { name, dishes: [ { name (required), price (NUMBER), ingredients (STRING) } ] } ] }
```

**Response handling** (`parseResponse()`, `GeminiMenuExtractorService.php:60`) maps HTTP/content failures to specific user-facing messages, all thrown as `RuntimeException`:

| Condition | Message |
| --- | --- |
| Missing API key (pre-flight) | "Gemini API key is not configured..." |
| Image file missing (pre-flight) | "Image file not found..." |
| HTTP 429 | "Gemini API rate limit reached..." |
| HTTP 400/403 | "Gemini API rejected the request. The API key may be invalid." (logged) |
| Other non-2xx | "Gemini API returned an error (status)..." (logged) |
| `finishReason === 'SAFETY'` | "The image was blocked by Gemini's safety filters..." |
| Empty `candidates` | "Gemini returned no results..." |
| Empty `text` part | "Gemini returned an empty response..." |
| Invalid JSON | "Could not parse the extracted menu data..." (logged) |
| Missing `categories` key | "Gemini response was missing expected data..." (logged) |

The successful text payload is `json_decode`d from `candidates.0.content.parts.0.text` and passed to `normalise()`.

**Normalization** (`normalise()`, `GeminiMenuExtractorService.php:112`) defends against malformed AI output: trims category/dish names and skips empties; coerces `price` to a rounded 2-decimal float only when numeric and `> 0` (else `null`); trims `ingredients` to `null` when blank. The guaranteed output shape is `['categories' => [['name' => string, 'dishes' => [['name','price','ingredients']]]]]`.

#### Import / confirmation (`import()`)

This is the only step that mutates the menu. It is owner-scoped (`$restaurant->menuScans()->findOrFail($id)`) and rejects scans that are not `completed`, already `imported` (the `result.imported` flag), or have no extracted categories (`MenuScanController.php:117-129`). It then validates the categories submitted from the frontend (these may be **edited** by the owner, not the raw scan), capped at `restaurant->category_limit` / `dish_limit` per category, with per-dish name/price/ingredients rules (`MenuScanController.php:132-139`).

Before writing it checks the incoming counts against the restaurant's *remaining* category/dish allowance and returns a `422` with a precise "you can add N more" message if exceeded. The write runs in `DB::transaction` with a `lockForUpdate()` re-check on `result.imported` to prevent concurrent double-imports (throws/catches an `already_imported` sentinel `RuntimeException`). It marks `result.imported = true`, then creates each `Category` (translatable `name` keyed by `restaurant->default_locale ?? 'ar'`, sequential `display_order`) and bulk `Dish::insert()`s all dishes with translatable JSON `name`/`ingredients`, `price`, `is_available = true`, and ordering (`MenuScanController.php:160-206`). Response reports created counts.

#### Test coverage

`tests/Feature/MenuOwner/MenuScanControllerTest.php` covers: scan returns `202` and pushes `ProcessMenuScan` (with faked `Queue`/`Storage`); the daily limit returns `429` and pushes nothing once 3 scans exist; import creates the expected categories/dishes from a `completed()` factory scan; import is idempotent (second import → `422`); and cross-owner isolation (another owner gets `404` on both status and import). The `MenuScanFactory::completed()` state seeds a realistic `result.categories` payload (`database/factories/MenuScanFactory.php:33`).

### 8.2 MediaService — Image Optimization & Media Library

`app/Services/Global/MediaService.php` is a reusable, project-agnostic pipeline combining **Intervention Image** (GD driver) and **Spatie Media Library**. It handles two responsibilities: pre-storage optimization of uploads parked in a per-user temp area, and promotion of those temp files into a model's media collection on form save.

#### Temp-upload flow

`storeTempUpload(UploadedFile $file, string $context, int $userId)` (`MediaService.php:39`):
1. Purges stale temp files (`purgeStaleTemps()`).
2. Optimizes the image using the context's preset.
3. Stores it as `storage/app/temp/{userId}/{uuid}.jpg` and returns `['key', 'original_size', 'optimized_size', 'saved_percent']` (human-readable sizes; percent saved).

This is invoked by `TempUploadController::store()` (`app/Http/Controllers/MenuOwner/TempUploadController.php`) — the deferred-upload pattern where the browser uploads, gets a `key`, and the actual form submit later references that key. The temp path is **per-user** (`tempPath($userId, $key)` → `temp/{userId}/{key}.jpg`), which is the security boundary.

#### Sync into a media collection

`sync(HasMedia $model, ?string $key, bool $delete, string $collection, string $mediaName)` (`MediaService.php:71`):
- If `key` is present and the file exists at the **current authenticated user's** temp path, it clears the collection and attaches the temp file with the given name.
- Else if `delete` is true, it clears the collection.
- Crucially, it resolves the path from `auth()->id()`, not the passed model's owner — so a user referencing another user's `key` finds nothing (per-user path isolation).

Consumers: `RestaurantController` (logo + cover_image, `RestaurantController.php:64-65`), `DishController` (dish images), `CategoryController` (category images), and `OnboardingController`.

#### Optimization presets

`optimize()` (`MediaService.php:112`) reads the upload, applies a fit mode, and saves a JPEG:
- `fit: 'cover'` → `$image->cover(w,h)` (crop to fill).
- `fit: 'contain'` (default) → `$image->scaleDown(w,h)` (scale down only, never upscale).
- A fixed `quality` saves at that JPEG quality; otherwise `saveCompressed()` steps quality down from 90 in increments of 5 until the file fits `max_kb` (or hits the floor of 30) (`MediaService.php:157`).

Presets live in `config/image-optimization.php` (resolved by `preset()`, falling back to `generic`):

| Context | Fit | Dimensions | Target |
| --- | --- | --- | --- |
| `logo` | contain | 300×300 | max 50 KB |
| `cover_image` | cover | 1920×600 | quality 80 |
| `dish` | cover | 800×600 | max 50 KB |
| `category` | contain | 500×500 | max 50 KB |
| `generic` | contain | 1200×1200 | max 200 KB |

#### Internals & housekeeping

- `scratchPath()` builds unique intermediate paths in `storage/app/temp`.
- `purgeStaleTemps()` deletes temp/scratch `.jpg` files older than `TEMP_TTL_SECONDS` (3600 s = 1 hour) across all user folders, best-effort (`MediaService.php:188`). The comment is explicit that cross-user safety comes from per-user paths, not this window.
- `ensureDir()` creates directories at `0755`; `humanSize()` formats bytes as B/KB/MB.

Note: because images are optimized to a single JPEG before storage, the app does **not** register Spatie media conversions (`registerMediaConversions`/`addMediaConversion` appear nowhere) — optimization happens up-front rather than via on-demand conversions.

#### Test coverage

`tests/Feature/MediaSyncServiceTest.php` verifies temp paths are user-scoped, `sync()` attaches a temp image for the owning user, `sync()` **ignores** a key belonging to another user (the key isolation property), and `sync(..., delete: true)` clears the collection.

### 9.5 User impersonation

Impersonation lets an admin log in as a menu owner to reproduce issues, using `lasserafn/laravel-impersonate`.

- **Routes**: registered by `Route::impersonate()` (`routes/web.php:119`) inside an `auth` + `owner.locale` + `throttle:mutations` group, producing the named routes `impersonate` (take) and `impersonate.leave`.

- **Redirects** (`config/laravel-impersonate.php`): taking impersonation redirects to `/dashboard`; leaving redirects to `/`. The original admin id is held in the `impersonated_by` session key.
- **Tests** (`tests/Feature/ImpersonateTest.php`) lock in the behavior: an admin impersonating a menu owner is redirected to `/dashboard` and authenticated as that owner; a menu owner impersonating another user gets 403; an admin impersonating themselves gets 403; a guest is redirected to login; and the impersonating user can `impersonate.leave` to be redirected to `/` and restored as the admin.

## 10. Security & Hardening

This section documents the defensive layer introduced/consolidated on the `hardening-cleanup` branch. It covers automatic IP abuse blocking, response security headers, CAPTCHA verification, layered rate limiting, encryption-at-rest for OAuth tokens, and SSRF/XSS-resistant URL validation for social links. The whole stack is exercised by `tests/Feature/Security/*`.

### 10.1 IP Abuse Guard & Auto-Blocking

The abuse system has three cooperating pieces: a service that records strikes and decides bans (`AbuseGuard`), a persistence model with an "active block" scope (`BlockedIp`), and a request middleware that enforces blocks (`BlockAbusiveIps`).

#### `app/Services/Global/AbuseGuard.php`

| Member | Purpose |
| --- | --- |
| `isBlocked(string $ip): bool` | Returns whether an IP is currently blocked. Trusted IPs short-circuit to `false`. The DB lookup (`BlockedIp::active()->where('ip', $ip)->exists()`) is cached for 60s under `abuse_guard:blocked:{ip}`. On a `QueryException` it **fails open** (logs a warning, returns `false`) so a DB outage never locks everyone out (AbuseGuard.php:29-43). |
| `block(string $ip, ?string $reason, ?DateTimeInterface $expiresAt): void` | `updateOrCreate` on `blocked_ips` keyed by `ip`, then busts the block cache (AbuseGuard.php:45-53). |
| `unblock(string $ip): void` | Deletes the row and busts the cache (AbuseGuard.php:55-60). |
| `recordViolation(string $ip): void` | Increments a per-IP strike counter (`abuse_guard:strikes:{ip}`) inside a rolling 60s window. When strikes reach the threshold it auto-bans the IP for 1 hour with reason `auto: sustained abuse` and clears the counter (AbuseGuard.php:62-77). Trusted IPs are exempt. |
| `isTrusted(string $ip): bool` | `in_array($ip, config('security.trusted_ips'), true)` (AbuseGuard.php:83-86). |

Key tuning constants (AbuseGuard.php:13-21):

| Constant | Value | Meaning |
| --- | --- | --- |
| `BLOCK_CACHE_TTL` | 60 (s) | How long an `isBlocked` result is cached. |
| `STRIKE_WINDOW` | 60 (s) | Rolling window for counting strikes. |
| `STRIKE_THRESHOLD` | 20 | Strikes within the window that trigger an auto-ban. Deliberately high so a busy shared NAT keeps headroom while a real flood (hundreds/min) trips instantly. |

Auto-ban is **driven by the rate limiters**, not by the per-request throttle alone — only high-volume limiters call `recordViolation` (see 10.4). A normal 429 does not by itself escalate to a ban.

#### `app/Models/BlockedIp.php`

Table `blocked_ips` (`database/migrations/2026_06_07_083607_create_blocked_ips_table.php`):

| Column | Type | Notes |
| --- | --- | --- |
| `id` | bigint PK | |
| `ip` | string, indexed | The blocked address. |
| `reason` | string, nullable | e.g. `auto: sustained abuse` or a manual reason. |
| `expires_at` | timestamp, nullable | `null` = permanent block. Cast to `datetime`. |
| `timestamps` | | |

The `scopeActive()` query scope selects rows where `expires_at IS NULL OR expires_at > now()`, so expired temporary bans are automatically ignored without a cleanup job (BlockedIp.php:33-39).

#### `app/Http/Middleware/BlockAbusiveIps.php`

Constructor-injects `AbuseGuard`. On each request: if the IP is blocked **and** the authenticated user is not an admin, it aborts with `403 "Your access has been temporarily blocked."` (BlockAbusiveIps.php:21-23). The admin bypass (`auth()->user()?->isAdmin()`) means a locked-out office can still reach the app as an admin.

This middleware is **appended** to the `web` group in `bootstrap/app.php:20-23`, deliberately running *after* `StartSession`/auth so the admin bypass can actually see the authenticated user (a prepended copy would run before the session boots and `auth()->user()` would always be `null`).

#### Tests — `tests/Feature/Security/BlockedIpTest.php`

Covers: blocked IP gets 403; admin is not blocked; expired blocks pass; no-block passes; `isBlocked`/`block`/`unblock` round-trip; 20 `recordViolation` calls auto-ban and persist the row; 2 violations do not ban.

### 10.2 Security Response Headers

`app/Http/Middleware/SecurityHeaders.php` is **prepended** to the `web` group (`bootstrap/app.php:20-21`) so it wraps every response. It sets (SecurityHeaders.php:20-28):

| Header | Value | Protects against |
| --- | --- | --- |
| `X-Frame-Options` | `DENY` | Clickjacking (framing). |
| `X-Content-Type-Options` | `nosniff` | MIME-type sniffing. |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Referrer leakage. |
| `Permissions-Policy` | `geolocation=(), microphone=(), camera=()` | Unwanted device-API access. |
| `Content-Security-Policy` | `frame-ancestors 'none'` | Clickjacking (modern equivalent of X-Frame-Options). |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Protocol downgrade — **only added when `$request->secure()`** so local HTTP isn't pinned to HTTPS. |

`tests/Feature/Security/SecurityHeadersTest.php` asserts each header is present, and specifically that HSTS is **omitted** on insecure requests and **present** on `https://` requests.

### 10.3 CAPTCHA Integration (Google reCAPTCHA)

#### Service — `app/Services/ThirdParty/Captcha.php`

Verifies a token against Google's siteverify endpoint (`https://www.google.com/recaptcha/api/siteverify`). Behavior (Captcha.php:11-37):

- If `config('services.recaptcha.enabled')` is false, returns `true` (fully disabled / no-op) — this is the default in dev/test.
- A blank token returns `false`.
- Posts `secret` + `response` (+ optional `remoteip`) as a form with a 3s connect / 5s total timeout. Any thrown exception is swallowed and treated as failure (`return false`).
- Succeeds only if the HTTP call succeeded **and** the JSON `success` flag is true.

`recaptcha` config (`config/services.php:53-57`): `site_key`, `secret`, and `enabled => filled(env('RECAPTCHA_SECRET'))` — i.e. CAPTCHA turns on automatically once a secret is configured.

#### Rule — `app/Rules/ValidCaptcha.php`

A `ValidationRule` that resolves `Captcha` from the container, calls `verify($value, Request::ip())`, and fails with the translated `auth.captcha` message if invalid (ValidCaptcha.php:15-20).

#### Where it's enforced

The rule is attached to the `g-recaptcha-response` field in two public Form Requests:

| Form Request | Endpoint |
| --- | --- |
| `app/Http/Requests/Auth/LoginRequest.php` (line 35) | Login / `get-started` POST. |

### 10.4 Rate Limiting

Named limiters are registered in `app/Providers/AppServiceProvider.php` via `configureRateLimiters()` (called from `boot()`). Every limiter is keyed by `user()->id ?: ip()` and shares a custom 429 response. The `autoBan` flag decides whether breaching the limit feeds `AbuseGuard::recordViolation()` (AppServiceProvider.php:36-67):

| Limiter | Limit / min | autoBan (feeds IP ban) | Rationale |
| --- | --- | --- | --- |
| `auth` | 5 | No | A 429 here is usually a legit double-submit/shared login; must not lock an IP out. |
| `contact` | 10 | No | Low-ceiling public form; throttled but not ban-worthy. |
| `mutations` | 120 | **Yes** | Authenticated write endpoints; sustained breaches = flooding. |
| `uploads` | 20 | **Yes** | File/scan uploads. |
| `public` | 120 | **Yes** | Public menu + CSRF-exempt tracking endpoints. |

Limiter-to-route mapping (`routes/auth.php`, `routes/web.php`):

| Limiter | Routes |
| --- | --- |
| `throttle:auth` | `POST get-started` (login), `GET auth/google` + callback, `PUT password` (auth.php:13-20). |
| `throttle:contact` | `POST /contact` (web.php:134). |
| `throttle:mutations` | Menu-owner mutation group (web.php:60), authenticated owner group (web.php:118), `POST /onboarding/advance` (auth.php:26). |
| `throttle:uploads` | `POST /temp-upload`, `POST /menu-scan`, `POST /menu-scan/{id}/import` (web.php:65,98,100). |
| `throttle:public` | `POST /{slug}/track-exit`, `POST /{slug}/track-whatsapp-order`, `GET /{slug}` (web.php:139-152). |

In addition to the named limiter, `LoginRequest` has its own per-(email|IP) throttle of 5 attempts using `RateLimiter::tooManyAttempts`/`hit`/`clear`, firing a `Lockout` event and surfacing `auth.throttle` once exceeded (LoginRequest.php:74-98). This is a second, credential-stuffing-specific lockout layered under `throttle:auth`.

The CSRF-exempt tracking endpoints (`*/track-exit`, `*/track-whatsapp-order`) are excluded from CSRF in `bootstrap/app.php:29-32` because they're fired via `navigator.sendBeacon` on page exit (after the session/token may have expired) and carry no sensitive data — `throttle:public` is what curbs their abuse instead.

#### Tests — `tests/Feature/Security/RateLimitTest.php`

Registers local `auth`/`mutations`/`uploads` limiters and asserts each returns 429 when hammered, that requests under the ceiling pass, and that repeated failed logins against the real `login` route are throttled (429 or a Breeze lockout message) while remaining a guest.

### 10.5 Encrypted-at-Rest Social (OAuth) Tokens

`app/Models/SocialAccount.php` casts `access_token` and `refresh_token` with Laravel's `encrypted` cast (SocialAccount.php:22-26), so OAuth tokens are encrypted transparently on write and decrypted on read.

The backfill migration `database/migrations/2026_06_01_100334_encrypt_existing_social_account_tokens.php` migrates any pre-existing plaintext tokens:

- `up()` chunks `social_accounts` (200 rows at a time), and for each `access_token`/`refresh_token` skips blank values and values that already decrypt cleanly, otherwise re-stores them with `Crypt::encryptString` — making it **idempotent / safe to re-run**.
- `down()` reverses it, decrypting back to plaintext.
- `isEncrypted()` detects current state by attempting `Crypt::decryptString` and catching `DecryptException` (migration lines 70-79).

### 10.6 SSRF / XSS-Resistant Social Link Validation

`app/Http/Requests/SocialLinkRequest.php` governs creating/updating a restaurant social link. Authorization is policy-driven (`can('create'|'update', RestaurantSocialLink)`). The `url` field is validated with `['required', 'url', 'max:255', 'regex:/^https?:\/\//i']` (SocialLinkRequest.php:23), which rejects dangerous schemes:

- `javascript:` URLs (XSS) — rejected by the `regex`.
- `data:` URLs (HTML/script injection) — rejected by the `regex`.
- Only `http`/`https` links are accepted; the user-facing message is `auth`-friendly ("Only http or https links are allowed.").

`platform` is constrained to an allow-list (`in:instagram,x,facebook,tiktok,whatsapp`).

#### Tests — `tests/Feature/Security/SocialLinkUrlTest.php`

Verifies `javascript:alert(1)` and `data:text/html,...` are rejected with a `url` error and never persisted, that an empty URL returns the required message, and that a normal `https://` URL is accepted and stored.

### 10.7 Trusted IP Configuration

`config/security.php` exposes a single `trusted_ips` array, parsed from the comma-separated `SECURITY_TRUSTED_IPS` env var (trimmed, blanks filtered). These IPs (office/monitoring/admin egress) are **never** rate-banned or blocked by `AbuseGuard`, preventing a shared NAT from locking legitimate users out of their own app.

### 10.8 Security Posture Checklist

| Control | Status | Mechanism |
| --- | --- | --- |
| Automatic IP blocking on sustained abuse | Yes | `AbuseGuard` strikes → `blocked_ips` (1h auto-ban) |
| Blocked-IP enforcement at the edge | Yes | `BlockAbusiveIps` middleware (403), admin bypass |
| Fail-open on DB failure (availability) | Yes | `isBlocked` catches `QueryException` |
| Trusted-IP allow-list (anti-self-lockout) | Yes | `config/security.php` `trusted_ips` |
| Clickjacking protection | Yes | `X-Frame-Options: DENY` + CSP `frame-ancestors 'none'` |
| MIME sniffing protection | Yes | `X-Content-Type-Options: nosniff` |
| HSTS on HTTPS | Yes | `Strict-Transport-Security` (secure requests only) |
| Referrer + Permissions policy | Yes | `Referrer-Policy`, `Permissions-Policy` |
| CAPTCHA on login | Yes | `ValidCaptcha` rule in `LoginRequest` |
| CAPTCHA on public contact form | Yes | `ValidCaptcha` rule in `ContactRequest` |
| Brute-force login lockout | Yes | `throttle:auth` (5/min) + per-email/IP lockout in `LoginRequest` |
| Rate limiting on contact | Yes | `throttle:contact` (10/min) |
| Rate limiting on uploads / scans | Yes | `throttle:uploads` (20/min, feeds auto-ban) |
| Rate limiting on public/scan beacons | Yes | `throttle:public` (120/min, feeds auto-ban) |
| Rate limiting on mutations | Yes | `throttle:mutations` (120/min, feeds auto-ban) |
| OAuth tokens encrypted at rest | Yes | `encrypted` casts + backfill migration |
| SSRF/XSS hardening on user URLs | Yes | `http(s)`-only regex in `SocialLinkRequest` |
| CSRF (with deliberate beacon exemptions) | Yes | Laravel default; `track-exit`/`track-whatsapp-order` exempt |
| Test coverage of the above | Yes | `tests/Feature/Security/*` (5 suites) |

#### Notable observations / loose ends

- `tests/Feature/Security/CaptchaRuleTest.php` references `App\Services\Captcha`, which does not exist — the live class is `App\Services\ThirdParty\Captcha`. This appears to be a stale import left from a refactor.
- `RateLimitTest` registers its own local limiters (`auth`/`mutations`/`uploads`) rather than relying solely on the app's real definitions, so it validates throttle *behavior* but not the exact production ceilings; the live ceilings live in `AppServiceProvider::configureRateLimiters()`.

## 11. Frontend, Views, Blade Components & Internationalization

Qayema's frontend serves **three distinct audiences**, each with its own layout, asset bundle, and i18n strategy: (1) the **marketing portal** (public landing/legal/contact pages), (2) the **auth flow** (login + onboarding), and (3) the **menu-owner dashboard** (the authenticated back-office). A fourth surface — the public guest menu — reuses the `menu/*` Blade/Alpine components. The stack is plain Blade + Alpine.js 3 + Tailwind CSS v3, bundled by Vite, with most heavy styling delivered through hand-written CSS files (not Tailwind utilities).

### 11.1 Asset Pipeline (Vite + Tailwind v3)

Two parallel asset systems coexist:

| Pipeline | Source | Where used | Notes |
|----------|--------|-----------|-------|
| **Vite-managed** | `resources/css/*`, `resources/js/app.js` | auth views, dashboard | Hashed, manifest-driven via `@vite([...])` |
| **Static `public/portal/*`** | hand-authored CSS/JS shipped in `public/` | portal pages | Loaded with `<link>`/`<script>` tags, cache-busted by file `mtime` |

**Vite config** (`vite.config.js`) declares six entry points: `resources/css/app.css`, `welcome.css`, `login.css`, `ui.css`, `dashboard.css`, and `resources/js/app.js`, with `refresh: true` for HMR.

**Tailwind config** (`tailwind.config.js`) scans `resources/views/**/*.blade.php` (plus pagination/compiled views), extends the `sans` font family with `Inter`, and loads the `@tailwindcss/forms` plugin. **PostCSS** (`postcss.config.js`) runs `tailwindcss` + `autoprefixer`.

The CSS files split by surface and are mostly bespoke (Tailwind utilities appear inline only in `menu/*` components and the dashboard's Alpine transition classes):

- `resources/css/app.css` (3 lines) — only the `@tailwind base/components/utilities` directives.
- `resources/css/ui.css` (~1031 lines) — the reusable `ui-*` design system (buttons `ui-btn-*`, fields `ui-field`/`ui-input`, etc.). Shared by auth + dashboard.
- `resources/css/dashboard.css` (~2327 lines) — the `sbl-*` sidebar shell, command palette (`cp-*`), help drawer (`hg-*`), toasts (`toast-*`), delete modal (`del-*`), language picker (`lang-*`).

- `public/portal/css/{app,layout/navbar,layout/footer,pages/landing,pages/contact}.css` — the portal master layout's stylesheet set.

**Supported languages** are declared once in `config/locales.php` (the file's header documents the "add a language" recipe). Fourteen locales, each with display name, flag emoji, and RTL flag:

| Code | Language | RTL | Code | Language | RTL |
|------|----------|-----|------|----------|-----|
| `en` | English | no | `ru` | Русский | no |
| `ar` | العربية | **yes** | `tr` | Türkçe | no |
| `fr` | Français | no | `hi` | हिन्दी | no |
| `de` | Deutsch | no | `zh` | 中文 | no |
| `es` | Español | no | `ja` | 日本語 | no |

**Translation file structure** — `lang/{code}/` ships three PHP array files per UI locale:

| File | Top-level keys | Consumed by |
|------|----------------|-------------|
| `auth.php` | `failed`, `throttle`, `google_only`, `captcha`, `login`, `brand` | login + brand panel |
| `menu_owner.php` | `nav`, `common`, `dashboard`, `statistics`, `restaurant`, `menus`, `categories`, `dishes`, `social_links`, `settings`, `qr_code`, `dropzone`, `profile`, `onboarding`, `menu_scan`, `help`, `search`, `language_switcher` | dashboard shell + all dashboard pages |
| `portal.php` | `nav`, `hero`, `problem`, `solution`, `features`, `how`, `stats`, `pricing`, `stories`, `faq`, `final`, `footer`, `contact` | portal pages |

All 14 locales have `auth.php` + `menu_owner.php`; **`portal.php` exists only for `ar` and `en`** (the public landing is bilingual). `lang/vendor/filament/*` holds Filament's bundled component translations for ~70 languages. Views consume strings with Laravel's `__()` helper (and `@js(__(...))` to inject translations into the Alpine `helpGuide()`/`cmdPalette()` data).

**Locale switching** — there is **no URL prefix**; the active locale lives in the session key `owner_locale`:

- `App\Http\Middleware\SetMenuOwnerLocale` (aliased `owner.locale` in `bootstrap/app.php`) reads `session('owner_locale', config('locales.default'))`, validates it against `config('locales.supported')`, and calls `app()->setLocale()`.
- `routes/web.php` defines two switch routes that write the session and redirect back to the referer (falling back to login/dashboard): `route('locale.switch', $locale)` (guest-accessible, persists through login — used by the portal navbar and login page) and `route('owner.locale.switch', $locale)` (auth-only — used by the dashboard language picker).

**RTL handling** — every server-rendered shell computes direction the same way: `$isRtl = in_array(app()->getLocale(), config('locales.rtl', []))` then emits `<html lang="{{ $locale }}" dir="{{ $isRtl ? 'rtl' : 'ltr' }}">` (see `sidebar-layout.blade.php:2`, `portal/.../master.blade.php:8`, `auth/login.blade.php:12`, `onboarding`). Arabic-aware fonts (Noto Kufi Arabic, El Messiri) are loaded in the auth/legal/portal heads. The `<x-auth.brand-panel>` receives `:is-rtl` to flip quote direction. Numeric/prefixed `ui.input` fields force `dir="ltr"` even inside an RTL page (input.blade.php:27,43).

**Two i18n models coexist:**
1. **Server-side (session-based `__()`)** — used by the dashboard, auth, portal, and contact pages. The chosen locale persists in the session and re-renders the whole page.
2. **Client-side bilingual (Alpine)** — used by the legacy landing path and the legal pages via `resources/views/partials/qayema-app.blade.php`, an Alpine `QayemaApp()` data function holding the entire en/ar copy tree inline. Its `setLang(l)` toggles `document.documentElement.lang`/`dir` in the browser without a round-trip (`x-text` bindings swap copy live). This is why the legal layout and the older welcome path embed full English+Arabic strings in JS rather than reading `lang/*` files.

### 12.2 Testing Setup

**Framework & harness.** Tests are **PHPUnit 11** (no Pest), split into two suites in `phpunit.xml`: `Unit` (`tests/Unit`) and `Feature` (`tests/Feature`). `tests/TestCase.php` is an empty extension of Laravel's `BaseTestCase` — there are no shared custom helpers; every test that touches the DB opts into `Illuminate\Foundation\Testing\RefreshDatabase` individually (only the two trivial `ExampleTest` cases and the `DishRequestTest` unit test skip it).

**Test environment** (forced in `phpunit.xml:20-34`): `APP_ENV=testing`, `DB_CONNECTION=sqlite` with `DB_DATABASE=:memory:` (in-memory; `RefreshDatabase` migrates fresh per test), `CACHE_STORE=array`, `SESSION_DRIVER=array`, `MAIL_MAILER=array`, `QUEUE_CONNECTION=sync`, `BCRYPT_ROUNDS=4` (fast hashing), and **`TELESCOPE_ENABLED=false`** / `PULSE_ENABLED=false` / `NIGHTWATCH_ENABLED=false`. reCAPTCHA is consequently disabled in tests (asserted in `CaptchaRuleTest`).

Several security tests register **throwaway routes in `setUp()`** to isolate middleware/limiters from app routing (e.g. `RateLimitTest` re-registers `auth`/`mutations`/`uploads` limiters and `/__throttle/*` routes; `SecurityHeadersTest` mounts `/__headers-test`; `BlockedIpTest` mounts `/__security-test`).

### 12.3 Seeders & Factories

**Seed order** (`database/seeders/DatabaseSeeder.php`): `SettingsSeeder → RestaurantTypeSeeder → TagSeeder → TemplateSeeder → TemplateTagSeeder → FeatureSeeder`, then a hard-coded **admin user** (`admin@admin.com` / `password`, role `Admin`).

A fresh `db:seed` produces:

- **Admin user** `admin@admin.com` (password `password`).
- **7 restaurant types** (`RestaurantTypeSeeder`): Restaurant, Café, Bakery, Food Truck, Bar, Cloud Kitchen, Home Cook (each with an emoji icon), `firstOrCreate` by slug.
- **110 tags** (`TagSeeder`): four categories × ~20–30 each — `cuisine`, `dietary`, `vibe`, `style`.
- **5 templates** (`TemplateSeeder`): `default`, `dark`, `elegant`, `minimal`, `bold`, all active.
- **Template↔tag links** (`TemplateTagSeeder`): each template tagged with 3 style/vibe tags (e.g. `default → light, minimal, classic`); silently skips missing rows.
- **5 features** (`FeatureSeeder`): limit features `dish_limit`/`category_limit`/`social_link_limit` and boolean add-ons `ordering`/`remove_branding` (translatable en/ar names). The free `default` template is bundled with `dish_limit=40`, `category_limit=10`, `social_link_limit=4`.
- `SettingsSeeder` is intentionally a **no-op** (settings table was removed; menu settings now live as `template_settings` JSON on `restaurants`).
